# RAG系统学习笔记

这是我阅读hello agent项目，rag章节的文档及源码的笔记。

## 一、RAG系统的技术分类

RAG（Retrieval-Augmented Generation，检索增强生成）的核心是"先检索、再生成"——在 LLM 回答之前，从外部知识库中取回相关信息注入 Prompt，以此弥补模型知识的静态性和有限性。

![RAG系统核心原理](https://raw.githubusercontent.com/yqh231/myNote/main/images/rag/8-5.png)

从技术演进角度，RAG系统可分为三代：

### 1.1 朴素RAG (Naive RAG)

最早期的形态，流程是直截了当的 **"检索-读取"（Retrieve-Read）** 模式。

- **检索**：基于传统关键词匹配算法（TF-IDF、BM25），通过词频和文档频率评估相关性
- **生成**：将检索到的文档原文拼接进 Prompt，不做任何处理直接交给 LLM
- **局限**：只能做字面匹配，无法理解语义；检索结果质量差时会引入噪声，反而干扰生成

### 1.2 高级RAG (Advanced RAG)

随着向量数据库和文本嵌入技术成熟，RAG在检索和生成的各环节引入优化。

- **检索**：从关键词匹配转向 **稠密嵌入（Dense Embedding）语义检索**，将文本映射为高维向量后计算相似度，能捕获同义词、近义表达等语义关系
- **生成**：引入查询重写（让查询更精准）、文档分块（控制上下文粒度）、重排序（对候选结果二次排序）等优化手段
- **进步**：检索质量显著提升，但各模块之间仍是硬编码的线性流程，灵活性不足

### 1.3 模块化RAG (Modular RAG)

当前主流方向。将系统拆分为可插拔、可组合的独立模块，适应多样化场景。

- **检索**：混合检索（向量+关键词+图谱）、多查询扩展（MQE）、假设文档嵌入（HyDE）等策略可自由组合
- **生成**：思维链推理、自我反思与修正、上下文压缩等技术
- **核心优势**：每一层可独立替换和优化——换嵌入模型不影响业务逻辑，换向量数据库不影响检索策略

HelloAgents 采用的就是模块化架构，其 RAG pipeline 结构为：

```
用户层：RAGTool 统一接口
  ↓
应用层：智能问答、搜索、管理
  ↓
处理层：文档解析（MarkItDown）→ 智能分块 → 向量化
  ↓
存储层：向量数据库（Qdrant）/ 文档存储（SQLite）
  ↓
基础层：嵌入模型 / LLM / 数据库连接
```

### 1.4 三代对比

| 维度 | 朴素RAG | 高级RAG | 模块化RAG |
|------|--------|--------|----------|
| 检索方式 | TF-IDF / BM25 | Dense Embedding | 混合检索 + MQE + HyDE |
| 文档处理 | 无/简单切分 | 固定窗口分块 | 结构感知智能分块 |
| 可扩展性 | 差 | 中等 | 各模块可插拔替换 |
| 典型问题 | 语义理解弱 | 流程僵化 | 复杂度较高 |

## 二、RAG系统的检索方式

检索是 RAG 的核心环节，直接决定了最终生成质量的上限。HelloAgents 实现了从基础到高级的多种检索策略。

### 2.1 基础检索方式

**关键词检索**：基于 TF-IDF 或 BM25 算法，按词频匹配文档。速度快、实现简单，适合精确术语查找，但无法处理同义词和语义变体。在 HelloAgents 中，工作记忆的轻量检索就采用了 TF-IDF 作为兜底方案。

**向量语义检索**：将查询和文档都转换为嵌入向量，通过余弦相似度等指标计算语义距离。能理解"机器学习"和"ML"是同一概念。这是 RAG 系统的主力检索方式，由 Qdrant 向量数据库承担。

**图谱检索**：通过知识图谱中的实体关系进行多跳推理。例如查询"张三的同事负责什么项目"，需要先找到张三，再沿"同事"关系找到人，再沿"负责"关系找到项目。纯向量检索无法完成这类关联推理，由 Neo4j 图数据库承担。

### 2.2 混合检索策略

实际系统中，单一检索方式各有盲区，因此 HelloAgents 对不同记忆类型采用了不同的混合策略：

| 记忆类型 | 混合方式 | 评分公式 |
|---------|---------|---------|
| 工作记忆 | TF-IDF向量 + 关键词匹配 | `(vec×0.7 + keyword×0.3) × 时间衰减 × 重要性权重` |
| 情景记忆 | 结构化过滤 + 向量检索 | `(vec×0.8 + 时间近因性×0.2) × 重要性权重` |
| 语义记忆 | 向量检索 + 图谱检索 | `(vec×0.7 + graph×0.3) × 重要性权重` |
| 感知记忆 | 模态专属向量检索 | `(vec×0.8 + 时间近因性×0.2) × 重要性权重` |

所有公式中的**重要性权重**统一为 `0.8 + importance × 0.4`，映射到 [0.8, 1.2] 区间，避免重要性过度干扰相似度排序。

关于排序中各因子的设计逻辑：
- **情景记忆强调时间近因性**：具体事件（"昨天的会议"）天然有时效性，越近的事件越可能被需要
- **语义记忆引入图谱分数**：抽象知识（"Python是解释型语言"）的价值在于概念间的关联，图谱能发现向量检索遗漏的隐含关系
- **工作记忆使用关键词兜底**：纯内存中数据量小，TF-IDF足以应对，且不依赖外部服务

### 2.3 多查询扩展 (MQE) 详解

#### 要解决的问题

用户的查询表述与知识库中的实际文档用词经常存在差异。例如用户问"如何学Python"，而文档标题是"Python入门指南"——语义一致但字面不同。如果只用一种表述去做向量检索，可能会遗漏本该匹配到的相关文档。这就是**词汇鸿沟（Lexical Gap）**问题。

#### 核心原理

MQE 的思路很直接：既然一种表述不够，那就用多种表述。具体做法是让 LLM 将原始查询改写为多个语义等价但用词不同的查询，对每个查询分别做向量检索，最后将所有结果合并去重。

#### 完整工作流程

```
原始查询: "如何学习Python"
         ↓
    ┌────────────────────────────────┐
    │  LLM 查询扩展                   │
    │  System: "你是检索查询扩展助手。  │
    │  生成语义等价或互补的多样化查询。  │
    │  使用中文，简短，避免标点。"      │
    │  User: "原始查询：如何学习Python  │
    │  请给出2个不同表述的查询，每行一个"│
    └────────────────────────────────┘
         ↓
扩展结果: ["Python入门教程",
          "Python学习方法"]
         ↓
并行检索: 原始查询 → 候选集A
         扩展查询1 → 候选集B
         扩展查询2 → 候选集C
         ↓
合并去重: 同一文档出现多次 → 保留最高分
         ↓
排序返回: 按分数降序取 top-k
```

#### 关键设计细节

**扩展数量**：默认生成 2 条扩展查询。加上原始查询共 3 条，在召回率和开销之间取得平衡。扩展太多会增加 LLM 调用和检索时间，太少则覆盖不足。

**Prompt 设计要点**：要求"简短、避免标点"是为了让扩展查询更接近检索 query 的风格，而不是生成完整的句子。完整句子反而会引入噪声词，降低向量匹配精度。

**容错机制**：如果 LLM 调用失败（超时、API 异常等），直接回退到原始查询，保证检索流程不中断。

**去重策略**：同一篇文档可能被不同扩展查询检索到，此时只保留最高的相似度分数，避免重复文档占据 top-k 名额。

#### 效果与适用场景

- 可提升召回率 30%-50%，对模糊查询和专业术语效果显著
- 额外开销是 1 次 LLM 调用（生成扩展查询）+ N 次向量检索，整体延迟增加可控
- 特别适合用户不确定"该怎么问"的探索性查询

#### MQE 的局限

MQE 本质上是在做**查询侧的同义改写**，它能解决"用词不同但意思相同"的问题，但无法解决查询和文档之间更深层的语义鸿沟——比如用户问的是疑问句，文档写的是陈述句，两者在向量空间中的分布本身就不同。这正是 HyDE 要解决的问题。

### 2.4 假设文档嵌入 (HyDE) 详解

#### 要解决的问题

向量检索的前提是"语义相似的内容在向量空间中距离相近"。但有一个反直觉的现象：**问题和它的答案在向量空间中往往并不相近**。

原因在于问题通常是疑问句（"什么是自注意力机制？"），而文档是陈述句（"自注意力机制通过计算序列中各位置间的关联权重..."）。两者的句式结构、用词模式都不同，导致嵌入向量的分布存在系统性偏差。这就是**查询-文档语义鸿沟（Query-Document Semantic Gap）**。

#### 核心原理

HyDE 的思路非常巧妙：**用答案找答案**。

既然问题和答案的向量距离远，那就先让 LLM 生成一个"假设性答案"，再用这个假答案的向量去检索知识库中的真实文档。假答案的内容不一定准确（甚至可能有错），但它的**表述形式**（陈述句）和**关键术语**与真实文档高度相似，因此在向量空间中更容易匹配到正确的文档。

#### 完整工作流程

```
用户问题: "Transformer的自注意力机制是什么？"
         ↓
    ┌──────────────────────────────────────┐
    │  LLM 生成假设文档                      │
    │  System: "根据用户问题，先写一段可能的   │
    │  答案性段落，用于向量检索的查询文档       │
    │  （不要分析过程）。"                    │
    │  User: "问题：Transformer的自注意力     │
    │  机制是什么？请直接写一段中等长度、       │
    │  客观、包含关键术语的段落。"              │
    └──────────────────────────────────────┘
         ↓
假设文档: "自注意力机制（Self-Attention）是Transformer
         架构的核心组件。它通过计算输入序列中每个位置与
         所有其他位置之间的注意力权重，使模型能够捕获
         长距离依赖关系。具体来说，输入会被映射为Query、
         Key、Value三个矩阵..."
         ↓
    对假设文档做向量编码（而非对原始问题编码）
         ↓
    用该向量在知识库中检索真实文档
         ↓
    返回与假设文档最相似的 top-k 真实文档
```

#### 为什么假答案不准确也能工作

这是理解 HyDE 最关键的一点。假设 LLM 对自注意力的描述有细节错误（比如把"softmax归一化"写成了"L2归一化"），但假答案中仍然包含了"自注意力"、"Transformer"、"Query/Key/Value"、"注意力权重"等**领域核心术语**。这些术语的嵌入向量与真实文档中的对应术语高度接近，足以引导检索系统找到正确的文档。

换句话说，HyDE 利用的不是假答案的**事实准确性**，而是它的**语言形式和术语分布**。

#### 关键设计细节

**Prompt 设计要点**：
- "不要分析过程"——避免 LLM 输出"让我来分析一下..."这类元话语，只要纯粹的答案段落
- "中等长度"——太短则术语覆盖不足，太长则引入噪声
- "客观、包含关键术语"——引导 LLM 生成百科风格的陈述，与知识库文档的风格最为接近

**容错机制**：如果 LLM 生成失败，HyDE 返回 None，系统自动回退到用原始查询做检索。

**额外开销**：比 MQE 更大——需要 1 次 LLM 调用生成假设文档（输出 token 更多），但只增加 1 次向量检索。

#### HyDE 特别有效的场景

- **专业领域查询**：用户可能不知道精确术语，但 LLM 能在假设答案中补充领域术语
- **跨语言检索**：用户用中文提问，假设答案中可能包含英文术语，恰好匹配到英文文档
- **复杂概念查询**：如"解释反向传播的链式法则"，假设答案展开后包含"梯度"、"偏导数"、"损失函数"等关联概念

#### HyDE 的局限

- 依赖 LLM 生成质量，如果 LLM 对查询领域完全不了解，假设答案可能偏离方向
- 对于简单的事实查询（"Python的最新版本号"），HyDE 的额外开销得不偿失
- 每次查询多调用一次 LLM，延迟和成本都会增加

### 2.5 MQE + HyDE 联合检索框架

HelloAgents 将两种策略整合进统一的 `search_vectors_expanded` 函数，通过参数开关自由组合。

#### 整体流程

```
原始查询 ──────────────────────────────→ 扩展列表 [原始查询]
              │                                    │
              ├── enable_mqe=True ──→ LLM改写 ──→ 追加 [扩展查询1, 扩展查询2]
              │                                    │
              └── enable_hyde=True ──→ LLM假设 ──→ 追加 [假设文档]
                                                   │
                                              去重（保留顺序）
                                                   │
                                         分配候选池（总池 = top_k × 4）
                                                   │
                                      每个扩展查询分得 总池/扩展数 的配额
                                                   │
                                         逐个查询 → 向量检索
                                                   │
                                      按 memory_id 聚合，保留最高分
                                                   │
                                         按分数降序 → 返回 top-k
```

#### 候选池分配机制

这是一个容易被忽略但很重要的设计。假设 `top_k=8`，`candidate_pool_multiplier=4`，则总候选池为 32 条。如果有 4 个扩展查询（原始 + 2条MQE + 1条HyDE），每个查询分配 8 条候选。

为什么要 4 倍扩大候选池？因为不同扩展查询可能检索到相同的文档，去重后实际数量会减少。如果不扩大候选池，去重后可能凑不够 top-k 条结果。

#### MQE 与 HyDE 的互补关系

| 维度 | MQE | HyDE |
|------|-----|------|
| 解决的问题 | 用词差异（同义表达） | 句式差异（问句 vs 陈述句） |
| 改变的对象 | 查询的**措辞** | 查询的**形式** |
| 检索向量数 | 多个（每个扩展各一个） | 1个（假设文档的向量） |
| LLM输出量 | 短（几个短句） | 长（一个段落） |
| 延迟开销 | 小 | 中等 |
| 对哪类查询提升大 | 模糊查询、口语化表述 | 专业查询、概念性问题 |

两者互不冲突，同时启用时覆盖面最广：MQE 从多个角度扩展查询用词，HyDE 从答案视角桥接语义鸿沟，形成互补。

#### 策略选择建议

| 场景 | 推荐策略 | 原因 |
|------|---------|------|
| 日常查询 | 基础检索 + MQE | MQE 开销小，收益明显 |
| 专业领域 | MQE + HyDE | HyDE 生成的领域术语能大幅提升精度 |
| 延迟敏感 | 仅基础检索 | 避免额外的 LLM 调用 |
| 高召回要求 | MQE + HyDE + 大候选池 | 候选池倍数建议 4x |
| 知识库较小 | 仅基础检索 | 文档少时基础检索已足够，扩展反而引入噪声 |

## 三、第三方数据库详解

HelloAgents 的记忆和 RAG 系统依赖三个第三方数据库，各司其职、互为补充。

![HelloAgents记忆与RAG系统架构图](https://raw.githubusercontent.com/yqh231/myNote/main/images/rag/8-2.png)

### 3.1 Qdrant — 向量数据库

**在系统中的角色**：所有需要语义相似度检索的场景都由 Qdrant 承担，是整个系统的检索引擎核心。

**服务对象**：情景记忆的向量检索、语义记忆的向量检索、感知记忆的多模态向量检索、RAG pipeline 的文档检索。

#### 基本概念

Qdrant 将数据组织为 **Collection（集合）**，每个 Collection 存储固定维度的向量及其关联的 **Payload（元数据）**。可以类比为：Collection 是表，向量是主键，Payload 是附加字段。

#### 连接方式

支持三种部署模式：
- **云服务（推荐）**：通过 URL + API Key 连接 Qdrant Cloud，免运维
- **本地 Docker**：`http://localhost:6333`，适合开发测试
- **内嵌模式**：纯本地文件存储，无需服务进程

HelloAgents 使用单例模式（`QdrantConnectionManager`）管理连接，避免重复创建客户端。

#### 核心配置

| 参数 | 说明 | 默认值 |
|------|------|-------|
| `vector_size` | 向量维度，必须与嵌入模型输出一致 | 384 |
| `distance` | 相似度度量方式 | cosine |
| `collection_name` | 集合名称 | hello_agents_vectors |

#### 在系统中的使用方式

**写入**：将文本通过嵌入模型转换为向量后，连同元数据（memory_type、user_id、timestamp、importance 等）批量写入 Qdrant。RAG pipeline 中还会额外标记 `is_rag_data=True` 和 `rag_namespace` 以实现数据隔离。

**检索**：传入查询向量，Qdrant 通过 HNSW 索引快速找到最相似的 top-k 个向量，同时支持基于 Payload 的条件过滤（如只搜索某个用户的数据、只搜索某种记忆类型）。

**数据隔离**：不同记忆类型和 RAG 命名空间通过 Payload 中的过滤字段实现逻辑隔离，共享同一个 Collection，避免维护过多集合。感知记忆例外——因为不同模态（文本/图像/音频）的向量维度不同，必须使用独立的 Collection。

#### 为什么选 Qdrant

- 原生支持 Payload 过滤，检索时可同时做语义匹配和条件筛选
- HNSW 索引在百万级数据下仍保持毫秒级响应
- 提供云服务，省去自建集群的运维成本

### 3.2 Neo4j — 图数据库

**在系统中的角色**：专门服务于语义记忆，负责存储和查询实体之间的关系网络（知识图谱）。

**解决的问题**：向量检索擅长"找相似的"，但无法处理"A和B是什么关系""从A出发经过哪些关系能到达C"这类结构化推理。

#### 基本概念

Neo4j 是属性图数据库，核心数据模型只有三个元素：
- **Node（节点）**：代表实体，如"张三"、"Python"、"机器学习"
- **Relationship（关系）**：连接两个节点，带类型和方向，如 `张三 -[擅长]-> Python`
- **Property（属性）**：节点和关系上的键值对，如 `{importance: 0.9, created_at: "2024-03-15"}`

#### 连接方式

- **Neo4j Aura 云服务（推荐）**：`neo4j+s://xxx.databases.neo4j.io`，免费层即可满足学习需要
- **本地 Docker**：`bolt://localhost:7687`

连接时配置连接池大小（默认50）和超时时间（默认60秒），避免高并发下连接耗尽。初始化时自动创建索引以加速查询。

#### 在系统中的使用方式

**写入**：当一条语义记忆被添加时，系统通过 NLP（spaCy，支持中英文）自动从文本中提取实体和关系，然后：
1. 为每个实体创建 Node，附带类型、名称、关联的记忆 ID 等属性
2. 为实体间的关系创建 Relationship，附带关系类型和置信度

例如文本"张三是Python开发者，专注于机器学习"会生成：
```
(张三:Person) -[是]-> (Python开发者:Role)
(张三:Person) -[专注于]-> (机器学习:Domain)
```

**检索**：语义记忆检索时，先从查询中提取实体，然后在图中进行多跳遍历，找到关联的记忆 ID，再结合向量检索的结果做混合排序。图检索的权重占 0.3，作为向量检索的补充。

**图检索独特价值的例子**：
- 查询"和张三相关的技术领域" → 图遍历 `张三 → 专注于 → 机器学习`，向量检索可能找不到这条间接关联
- 查询"谁擅长Python" → 图遍历所有指向Python节点的"擅长"关系，直接命中

#### 为什么选 Neo4j

- Cypher 查询语言表达力强，多跳关系查询直观高效
- 属性图模型天然适合知识图谱场景
- Aura 云服务降低了使用门槛

---

### 3.3 SQLite — 文档存储

**在系统中的角色**：作为结构化数据的持久层，存储记忆的完整元信息，支持精确的条件查询和统计分析。

**解决的问题**：Qdrant 存的是向量和简要元数据，Neo4j 存的是实体关系，但记忆的完整结构化信息（ID、内容全文、类型、重要性、时间戳、会话ID、各种元数据字段）需要一个可靠的关系型存储来管理。

#### 基本概念

SQLite 是嵌入式关系数据库，无需独立服务进程，数据存储在单个文件中。适合中小规模数据的持久化。

#### 连接方式

直接指定数据库文件路径即可，如 `./memory_data/memory.db`。使用线程本地连接（thread-local）保证多线程安全，通过单例模式避免同一文件被重复打开。初始化时自动创建表结构和索引。

#### 在系统中的使用方式

**写入**：情景记忆在添加时，除了将向量存入 Qdrant，还会将完整的记忆记录（ID、内容、类型、重要性、时间戳、元数据 JSON 等）写入 SQLite。

**查询**：支持多维度条件组合查询：
- 按用户过滤：`WHERE user_id = ?`
- 按记忆类型：`WHERE memory_type = 'episodic'`
- 按时间范围：`WHERE timestamp BETWEEN ? AND ?`
- 按重要性阈值：`WHERE importance >= ?`
- 复合排序：`ORDER BY importance DESC, timestamp DESC`

**统计**：提供记忆数量统计、类型分布、时间分布等分析能力，用于生成学习报告和系统监控。

**结构化预过滤**：在情景记忆的检索流程中，SQLite 先做第一轮条件过滤（时间范围、重要性等），缩小候选集后再交给 Qdrant 做向量检索，兼顾精确性和效率。

#### 为什么选 SQLite

- 零配置、零运维，无需额外部署服务
- 单文件存储，便于备份和迁移
- 适合中小规模的结构化查询场景
- 与 Python 原生集成，无额外依赖

---

### 3.4 三库协作关系

三个数据库在系统中不是互相替代，而是各取所长、分工协作：

```
用户查询
  ↓
MemoryManager 统一调度
  ├── 工作记忆: 纯内存（无外部数据库）
  ├── 情景记忆: SQLite(结构化过滤) + Qdrant(语义检索)
  ├── 语义记忆: Qdrant(语义检索) + Neo4j(关系推理)
  ├── 感知记忆: Qdrant(多模态向量检索)
  └── RAG检索:  Qdrant(文档向量检索)
```

| 数据库 | 存什么 | 查什么 | 服务于 |
|-------|-------|-------|-------|
| **Qdrant** | 嵌入向量 + 简要元数据 | 语义相似度 | 所有需要语义匹配的场景 |
| **Neo4j** | 实体节点 + 关系边 | 多跳关联推理 | 语义记忆的知识图谱 |
| **SQLite** | 完整记忆记录 | 条件过滤 + 统计 | 情景记忆的结构化查询 |

这种设计的好处是每个数据库只做自己最擅长的事，且任一组件都可以独立替换（如用 Milvus 替换 Qdrant，用 ArangoDB 替换 Neo4j），不影响整体架构。

## 四、检索排序算法深度分析

检索只是找到候选文档，而**排序决定了用户最终看到什么**。一个好的排序算法需要综合考虑语义相关性、时间因素、重要性等多个维度。以下从 HelloAgents 的实际实现出发，结合业内常用做法进行分析。

### 4.1 HelloAgents 四种记忆的排序算法

#### 工作记忆：指数时间衰减

工作记忆面对的是"当前对话"场景，信息的新鲜度极为重要。其排序分为三步：

**第一步：基础相关性**
- 优先使用 TF-IDF 向量相似度（权重 0.7）+ 关键词匹配分数（权重 0.3）
- 如果 TF-IDF 不可用，直接退化为纯关键词匹配

**第二步：指数时间衰减**
- 采用指数衰减函数：`decay = decay_factor ^ (hours_passed / 6)`
- 每 6 小时衰减一个周期，最低保持 0.1（不会完全归零）
- 这是一条**陡峭的衰减曲线**，几小时前的信息就会被显著降权

**第三步：重要性调制**
- 重要性映射到 [0.8, 1.2] 区间：`weight = 0.8 + importance × 0.4`
- 作为乘法因子，最多让分数浮动 ±20%

设计意图：工作记忆是"临时便签"，越新的越重要，老信息快速淡出。指数衰减比线性衰减更符合人类短期记忆的遗忘规律。

#### 情景记忆：倒数时间衰减

情景记忆存储的是具体事件，时间跨度从几天到几个月，衰减曲线需要比工作记忆更平缓。

**基础相关性** = 向量相似度 × 0.8 + 时间近因性 × 0.2

**时间近因性**采用倒数衰减：`recency = 1 / (1 + age_days)`

这条曲线的特点：
- 当天的事件：recency = 1.0（满分）
- 1天前：recency = 0.5
- 6天前：recency ≈ 0.14
- 29天前：recency ≈ 0.03

相比工作记忆的指数衰减，倒数衰减的尾部更长——几周前的事件仍有微弱的近因性得分，不会被完全忽略。

**最终得分** = 基础相关性 × (0.8 + importance × 0.4)

设计意图：事件有"越近越相关"的倾向，但不如工作记忆那么极端。向量相似度占主导（0.8），时间只是辅助调节。

#### 语义记忆：向量+图谱双通道融合

语义记忆是四种记忆中排序最复杂的，因为它同时使用了向量检索和图谱检索两个通道。

**向量通道**：Qdrant 返回余弦相似度分数

**图谱通道**：Neo4j 返回的图相关性分数本身也是一个复合评分：
- 实体匹配度 × 0.6 — 查询实体和记忆实体的交集比例
- 实体密度 × 0.2 — 该记忆包含的实体数量（归一化到 [0,1]，上限 10 个）
- 关系密度 × 0.2 — 该记忆包含的关系数量（归一化到 [0,1]，上限 5 个）

**双通道融合**：`base_relevance = vector_score × 0.7 + graph_score × 0.3`

**Softmax 概率归一化**：语义记忆在返回结果时，额外对所有候选的 combined_score 做了 softmax 归一化，将绝对分数转换为概率分布。这意味着结果不仅有排序，还有"这条结果有多大置信度"的语义——当所有候选分数都很低时，概率会趋于均匀分布，暗示"没有明确的最佳答案"。

**内容去重**：语义记忆在合并两个通道结果时，通过 content hash 做了严格的内容级去重（而非仅 ID 去重），避免同一段知识因为同时被向量和图谱命中而重复出现。

**最低阈值过滤**：综合分数低于 0.1 的结果直接丢弃，避免低质量结果进入最终排序。

设计意图：知识没有时效性（"Python是解释型语言"不会过期），所以不引入时间衰减。图谱补充了向量无法捕捉的结构化关系，softmax 提供了置信度信息。

#### 感知记忆：与情景记忆一致

感知记忆的排序算法与情景记忆完全相同：向量相似度 × 0.8 + 倒数时间近因性 × 0.2，再乘以重要性权重。不同之处在于它按模态（text/image/audio）使用独立的向量集合，确保不同维度的向量不会混淆。

### 4.2 四种排序算法对比

| 维度 | 工作记忆 | 情景记忆 | 语义记忆 | 感知记忆 |
|------|---------|---------|---------|---------|
| 相关性来源 | TF-IDF + 关键词 | 向量相似度 | 向量 + 图谱 | 向量相似度 |
| 时间衰减 | 指数衰减（6h周期） | 倒数衰减（天级） | 无 | 倒数衰减（天级） |
| 衰减速度 | 快（小时级失效） | 中（周级缓降） | 不衰减 | 中（周级缓降） |
| 重要性权重 | [0.8, 1.2] | [0.8, 1.2] | [0.8, 1.2] | [0.8, 1.2] |
| 概率输出 | 无 | 无 | Softmax归一化 | 无 |
| 去重机制 | 无（内存量小） | ID去重 | Content Hash | ID去重 |
| 最低阈值 | 无 | 无 | 0.1 | 无 |

### 4.3 与业内常用排序方法的对比

HelloAgents 的排序属于**多因子加权融合（Multi-Factor Weighted Fusion）**，这是 RAG 系统中最常见的排序范式之一。以下对比业内几种主流方法：

#### (1) 倒数排名融合 (RRF - Reciprocal Rank Fusion)

RRF 是混合检索中最经典的排序方法，被 Elasticsearch、LangChain 等广泛采用。

公式：`RRF_score = Σ 1/(k + rank_i)`，其中 k 通常取 60

原理：不关心各检索通道返回的绝对分数（因为不同检索方式的分数不可比），只关心**排名位置**。每个通道中排名靠前的文档获得更高的 RRF 分数。

**对比 HelloAgents**：HelloAgents 直接对向量分数和图谱分数做加权求和（0.7/0.3），这要求两个通道的分数在同一量纲内。Qdrant 返回的余弦相似度天然在 [0,1] 范围内，图谱分数也被归一化到 [0,1]，所以加权求和是可行的。但如果混合的检索通道分数量纲差异大（如 BM25 分数 vs 余弦相似度），RRF 会更稳健。

#### (2) 交叉编码器重排序 (Cross-Encoder Reranking)

思路：用一个专门的交叉编码器模型（如 BGE-Reranker、Cohere Rerank）对初筛结果做二次精排。交叉编码器同时输入 query 和 document，能捕获两者之间更精细的交互特征。

典型流程：向量检索召回 top-100 → 交叉编码器重排 → 返回 top-10

**对比 HelloAgents**：HelloAgents 没有使用重排序模型，排序完全基于规则公式。优点是无额外模型开销、延迟低、可解释性强；缺点是排序精度不如交叉编码器。对于需要极高精度的生产系统，在 HelloAgents 的多因子融合之后再加一层 Reranker 是常见的增强方案。

#### (3) 学习排序 (Learning to Rank, LTR)

用机器学习模型（如 LambdaMART、神经网络）学习排序函数。训练数据通常来自用户的点击行为或人工标注。

**对比 HelloAgents**：HelloAgents 的权重（0.7/0.3、0.8/0.2 等）是人工设定的经验值。LTR 可以从数据中自动学习这些权重，在有充足训练数据的场景下效果更好。但 LTR 需要标注数据和训练流程，对于框架级的通用设计来说，人工权重是更务实的选择。

#### (4) 多阶段排序 (Multi-Stage Ranking)

生产级 RAG 系统通常采用漏斗式的多阶段排序：

```
召回（Retrieval）→ 粗排（Pre-ranking）→ 精排（Ranking）→ 重排（Reranking）
    ~1000条            ~100条             ~20条            ~5条
```

每个阶段使用更重的模型、更少的候选。HelloAgents 的流程可以映射为：向量/图谱检索（召回）→ 多因子加权融合（粗排+精排）→ 阈值过滤（简单的重排）。如果要向生产级演进，可以在融合之后插入交叉编码器精排。

### 4.4 排序设计的核心权衡

| 权衡维度 | HelloAgents的选择 | 替代方案 |
|---------|-----------------|---------|
| 分数融合 vs 排名融合 | 加权分数融合（要求量纲统一） | RRF（量纲无关，更通用） |
| 规则权重 vs 学习权重 | 人工经验值 | LTR 从数据学习 |
| 单阶段 vs 多阶段 | 单阶段融合+阈值过滤 | 召回→粗排→精排→重排 |
| 时间衰减模型 | 指数/倒数（按场景区分） | 对数衰减、阶梯衰减 |
| 重要性作用方式 | 乘法因子 [0.8,1.2] | 加法因子、独立特征 |

HelloAgents 的排序设计在**简洁性和有效性之间取得了好的平衡**：四种记忆共享统一的重要性权重范围 [0.8, 1.2]，但各自根据数据特性选择不同的时间衰减模型和相关性计算方式。这种"统一框架+差异化策略"的设计，既降低了系统复杂度，又保证了各场景的排序质量。

## 五、知识总结

![Hello-agents第八章知识总结](https://raw.githubusercontent.com/yqh231/myNote/main/images/rag/8-11.png)

1. **RAG 从技术角度分三代**：朴素RAG（关键词匹配）→ 高级RAG（语义检索+流程优化）→ 模块化RAG（可插拔组件+策略组合）
2. **检索方式有三个层次**：基础检索（关键词/向量/图谱）→ 混合检索（多种方式加权融合）→ 高级策略（MQE解决用词差异，HyDE桥接语义鸿沟）
3. **三个数据库各司其职**：Qdrant 负责"找相似的"，Neo4j 负责"找相关的"，SQLite 负责"管记录的"
4. **排序算法的核心设计**：统一的重要性权重框架 [0.8, 1.2] + 差异化的时间衰减模型（工作记忆指数衰减、情景/感知倒数衰减、语义记忆不衰减）+ 语义记忆特有的向量-图谱双通道融合和 softmax 概率输出
5. **业内排序范式**：多因子加权融合（HelloAgents采用）、RRF排名融合（量纲无关）、交叉编码器重排序（高精度）、学习排序LTR（数据驱动），生产系统通常采用多阶段漏斗式排序
