## 注意力机制详解：原理、实现与工程实践

> 本文是我学习从0构建一个大模型的笔记，围绕注意力机制的核心设计动机展开：注意力机制是什么、它解决了什么问题、在实际中有哪些应用、Q/K/V 各自承担什么角色、缩放因子的数学原因、mask 必须位于 softmax 之前的原因、多头机制的表达力来源，以及工业界在此基础上做了哪些工程改造。
>
> 参考材料：《Build a Large Language Model (From Scratch)》第 3 章

## 1. 为什么 Transformer 需要注意力机制

在 RNN 时代，模型按时间顺序一步步处理 token。它的问题不是"不能看历史"，而是：

- **历史信息需要逐步传递**：离得越远，路径越长，信息越容易衰减。
- **并行性差**：时间步之间强依赖，训练和推理都慢。
- **长距离依赖建模困难**：一句话开头的词，想影响句尾的表示，需要穿过很长的计算链。

注意力机制把"顺序传递信息"改造成"按需检索信息"：

- 当前 token 不需要等待中间状态层层转发；
- 它可以**直接和序列中所有 token 建立相关性**；
- 谁重要，就给谁更大的权重。

所以，注意力最本质的作用不是"聚焦"，而是：

> **为每个位置动态构造一个与当前任务相关的上下文表示。**

也就是说，输入的每个 token 表示不再只是"它自己"，而会变成"它自己 + 它认为重要的其他 token 的信息"。

### 注意力机制具体能做什么

理解了注意力的本质之后，可以看看它在实际应用中的几个典型作用：

**1. 解决长距离依赖问题**

在处理 "The cat, which sat on the mat and played with the yarn for hours, was happy" 这样的长句时，注意力机制能让 "was" 直接关注到远处的 "cat"，正确建立主谓关系，而不需要像 RNN 一样逐词传递信息。

**2. 捕捉上下文语义**

"bank" 这个词在 "river bank" 和 "bank account" 中含义完全不同。注意力机制让每个词根据周围上下文动态调整自己的表示，从而实现精准的语义消歧。

**3. 实现信息的选择性聚焦**

在机器翻译中，翻译目标语言的某个词时，模型不需要均匀关注源语言的所有词，而是通过注意力权重聚焦到最相关的源词上。比如翻译 "I love you" 为 "我爱你" 时，生成 "爱" 的时候会主要关注 "love"。

**4. 支持并行计算**

与 RNN 的顺序计算不同，注意力机制让序列中所有位置可以同时计算彼此的关联，这使得 Transformer 能充分利用 GPU 的并行能力，极大提升训练效率。

**5. 泛化到多种模态**

注意力机制不仅用于文本。在计算机视觉（ViT）中，图像被切成 patch 后也通过注意力建模 patch 之间的关系；在语音识别中，注意力帮助模型定位关键帧；在多模态任务中，注意力实现了文本、图像、音频之间的跨模态对齐。

简单来说，注意力机制是当前几乎所有最先进 AI 模型的核心引擎。理解了它，就理解了 Transformer 的灵魂。

## 2. 用"检索系统"理解 Q / K / V

在展开公式之前，先建立一个核心直觉：

> **Attention 本质上是在做内容寻址（content-based retrieval）。**

可以把一段序列想成一个小型内存库：

- 每个 token 都会生成一个 **Key**：表示"我身上有什么可被匹配的特征"
- 每个 token 也会生成一个 **Value**：表示"如果你选中我，我要贡献什么信息"
- 当前正在更新的那个 token 会生成一个 **Query**：表示"我现在想找什么信息"

于是就有了一个自然流程：

1. 用 `Query` 去和所有 `Key` 做匹配，得到相关性分数；
2. 把分数归一化成概率分布；
3. 按这个概率分布对所有 `Value` 做加权求和；
4. 得到新的上下文向量 `context`。

这比"固定窗口"或者"纯顺序传递"灵活得多，因为：

- 相关性不是手工规定的，而是**数据驱动学出来的**；
- 同一个 token 在不同上下文里，可以检索完全不同的信息；
- 这种检索是可微的，因此能端到端训练。

---

## 3. 注意力机制的递进式实现路径

理解注意力机制，最好的方式是按照从简到完整的路径逐步构建：

- **第一步**：无可训练参数的简化 attention — 理解核心计算流程
- **第二步**：把单个 query 推广到整段序列 — 实现批量处理
- **第三步**：引入可训练的 `W_q / W_k / W_v` — 获得真正的学习能力
- **第四步**：加入 causal mask 与 dropout — 满足自回归和正则化需求
- **第五步**：扩展到 multi-head attention — 实现多视角信息检索

这条递进路径清晰展示了 Transformer attention 并非凭空设计，而是从最基本的"相似度加权平均"一步步演化而来。

## 4. 无参数版 self-attention：理解基本流程

我们先从一个**没有可训练权重**的简化版本开始。这个版本不会出现在生产模型中，但它完整展示了 attention 的核心计算流程。

假设输入序列是：

\[
X = [x^{(1)}, x^{(2)}, ..., x^{(T)}], \quad x^{(i)} \in \mathbb{R}^{d}
\]

如果我们想更新第 \(i\) 个 token，就把它当作 query：

\[
q^{(i)} = x^{(i)}
\]

然后它和每个输入做点积，得到 attention score：

\[
\omega_{ij} = q^{(i)} \cdot x^{(j)}
\]

再用 softmax 归一化：

\[
\alpha_{ij} = \text{softmax}(\omega_{ij})
\]

最后求加权和：

\[
z^{(i)} = \sum_{j=1}^{T} \alpha_{ij} x^{(j)}
\]

### 这个版本最重要的启发

- **attention 的核心不是复杂网络，而是"分数 + 归一化 + 加权和"**
- 上下文向量 `z^(i)` 不是简单复制 `x^(i)`，而是一个融合了全局信息的新表示
- 如果某个 token 和当前 query 更相关，它对最终结果贡献更大

### 为什么还不够

这个版本的问题在于：

- Query 和被匹配对象是同一个表示空间；
- "匹配特征"和"输出内容"没有分离；
- 模型没有能力学习"我应该根据什么找"和"找到后该拿出什么"。

这就是后面引入 Q/K/V 线性映射的根本原因。

## 5. 真正的 self-attention：为什么一定要把 Q、K、V 分开

在真实 Transformer 中，输入不会直接拿来互相点积，而是先经过三组线性变换：

\[
Q = XW_Q, \quad K = XW_K, \quad V = XW_V
\]

于是 attention 变成：

\[
\text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right)V
\]

### 这三者各自负责什么

- **Query**：当前 token 想找什么
- **Key**：当前 token 具备哪些可匹配特征
- **Value**：当前 token 被选中后要贡献什么内容

### 为什么这比"直接拿输入互相点积"强很多

因为模型终于获得了三种独立的自由度：

- **检索目标可以学**：不是所有相似都重要
- **匹配空间可以学**：语法相关、语义相关、位置相关都可以编码进去
- **输出内容可以学**：被关注不等于原样输出，可以是另一个变换空间里的信息

这背后一个非常关键的思想是：

> **"如何匹配"与"匹配后输出什么"是两件不同的事。**

例如，一个词在语义上可能与另一个词强相关，但最终输出时模型更想提取的是"句法角色"而不是"词面意义"。Q/K/V 分离使这种行为成为可能。

## 6. 为什么要除以 \(\sqrt{d_k}\)

这一步在教材实现里很容易被当成"公式规定"，但其实它是训练稳定性的关键。

### 直觉解释

如果 `query` 和 `key` 的维度很大，那么它们点积的数值方差会随着维度变大而增大。分数一旦变得很大：

- softmax 会过度尖锐；
- 某一个位置概率接近 1，其它位置接近 0；
- 梯度会变小，训练不稳定。

因此需要做缩放：

\[
\frac{QK^T}{\sqrt{d_k}}
\]

这样做的效果是把 logits 的量级控制在更适合 softmax 的范围里。

### 工程上怎么理解

你可以把它理解成：

- 不缩放：模型太容易"自信过头"
- 缩放后：softmax 更平滑，梯度更健康

这就是为什么在从"手工计算 attention score"过渡到"真正 self-attention"的实现中，必须加入 `d_k**0.5` 缩放因子。

---

## 7. 为什么 softmax 比"直接归一化求和"更合理

在学习 attention 的过程中，通常会先看到一个简单归一化方案，再切换到 softmax。这里值得重点记住两点：

### 7.1 softmax 不是为了凑概率，而是为了获得更好的优化性质

直接除以总和虽然也能让权重和为 1，但它：

- 对负数和极值不够稳健；
- 缺乏 softmax 那种"突出相对差异"的特性；
- 数值稳定性较差。

### 7.2 真正训练时应尽量使用框架提供的稳定实现

在实际实现中也需要特别注意这一点：

- 自己手写 `exp(x) / exp(x).sum()` 容易溢出；
- 实际应使用 `torch.softmax` 或更高层的 fused attention 接口。

这也是工程代码和教学代码的一个重要分界线：

> **教学代码强调"看见公式"，工程代码强调"数值稳定 + 内核效率 + 设备兼容"。**

## 8. 矩阵化之后，attention 其实是一种批量检索

单个 query 的形式很好理解，但真正高效的实现一定是矩阵化的。

对于整段序列：

- `Q` 形状通常是 `(T, d_k)`
- `K` 形状通常是 `(T, d_k)`
- `V` 形状通常是 `(T, d_v)`

则：

\[
S = QK^T \in \mathbb{R}^{T \times T}
\]

这里的 \(S_{ij}\) 表示"第 \(i\) 个 token 对第 \(j\) 个 token 的关注分数"。

归一化后得到注意力矩阵：

\[
A = \text{softmax}\left(\frac{S}{\sqrt{d_k}}\right)
\]

再与 `V` 相乘：

\[
Z = AV
\]

### 这个矩阵形式说明了什么

- attention 不是局部算子，而是**序列所有位置之间的两两交互**
- 每一行都代表一个 token 的"检索分布"
- 这也是为什么 attention 的时间和显存复杂度都会和 \(T^2\) 相关

这点在后面的工业实践部分非常关键，因为几乎所有 attention 优化，本质上都在和这个 \(T \times T\) 矩阵较劲。

## 9. Causal Attention：为什么 Decoder 必须遮住未来

对于 GPT 这类自回归语言模型，训练时我们虽然一次把整段序列都送进网络，但在预测第 \(t\) 个 token 时，**理论上只能看见 1 到 \(t\) 的历史**，不能偷看未来 token。

如果不加 mask：

- 第 3 个位置可能直接看见第 4、5、6 个 token；
- 模型训练时"作弊"，推理时却拿不到这些未来信息；
- 训练分布和推理分布不一致，模型学到的是错误问题。

因此 decoder-only 模型必须加因果掩码：

\[
M_{ij} =
\begin{cases}
0, & j \le i \\
-\infty, & j > i
\end{cases}
\]

然后：

\[
A = \text{softmax}\left(\frac{QK^T + M}{\sqrt{d_k}}\right)
\]

### 为什么 mask 要加在 softmax 之前

这一点非常重要，值得单独强调。

如果在 softmax **之后** 才把未来位置清零，会出现两个问题：

- 每一行的权重和不再等于 1；
- 需要额外的重新归一化步骤。

而如果在 softmax **之前** 把非法位置设为 \(-\infty\)：

- softmax 后这些位置自然变成 0；
- 合法位置自动重新分配概率质量；
- 数学上更干净，工程上也更高效。

### 这其实不是小技巧，而是标准实现范式

工业界基本都会采用：

- `masked_fill(mask, -inf)`
- 然后再 `softmax`

因为这和 fused kernel、FlashAttention、PyTorch SDPA 的实现方式也更一致。

## 10. 为什么 `register_buffer` 很重要

`CausalAttention` 中 mask 的典型处理方式是：

- 在初始化时生成上三角矩阵；
- 用 `register_buffer` 注册为 buffer；
- 前向时按当前 `num_tokens` 截取。

这背后体现的是一个非常典型的 PyTorch 工程习惯：

### buffer 的意义

它不是参数，但又是模型状态的一部分：

- **不参与梯度更新**
- **会跟随模型一起搬到 GPU / CPU**
- **会进入 state_dict**（除非特殊设置）

对于 mask 这种"需要持久保存、设备对齐、但不训练"的张量，buffer 是最合理的容器。

如果不用 buffer，而是每次 forward 现算：

- 会增加重复开销；
- 容易出现 device 不一致；
- 代码也更乱。

---

## 11. Dropout 在 attention 里放哪儿

教材中把 dropout 放在 `attn_weights` 上，这也是最常见做法之一。

即：

1. 先算 score
2. 再做 mask
3. 再做 softmax
4. 对 attention weights 做 dropout
5. 最后乘 `V`

### 它的含义是什么

这不是把某些 token 从输入里删掉，而是：

> **在训练时，随机打断部分注意力连接，防止模型过度依赖某几条固定关联。**

### 一个现实提醒

在现代大模型预训练中：

- attention dropout 往往设得很小；
- 有些大模型甚至直接设为 0；
- 大规模训练时，更依赖数据规模、权重衰减、学习率调度和归一化策略来稳住训练。

所以它是一个"经典但不一定总是重度使用"的组件。

另外，推理时 dropout 必须关闭。

## 12. 从单头到多头：为什么不是一个 attention 做到底

如果只有单头 attention，那么模型在每一层、每个位置只能形成一种检索模式。

这会带来限制：

- 它可能更关注语义相似性，但忽略句法依赖；
- 或者擅长近距离搭配，却不擅长长距离指代；
- 一个头的表示子空间容量有限。

Multi-Head Attention 的思想是：

- 先把 `d_model` 投影到多个子空间；
- 每个 head 在自己的子空间里独立做 attention；
- 最后把多个 head 的结果拼接起来，再做一次输出投影。

公式上可以理解为：

\[
\text{head}_i = \text{Attention}(Q_i, K_i, V_i)
\]

\[
\text{MHA}(X) = \text{Concat}(\text{head}_1, ..., \text{head}_h) W_O
\]

### 多头真正带来的不是"更多参数"，而是"多种关系视角"

一个常见误解是：多头只是把模型做大。

更准确地说，多头的价值在于：

- 不同头可以学习不同类型的依赖模式；
- 同一位置可以同时检索不同语义子空间的信息；
- 模型更容易形成分工。

当然，现实中也并不是每个头都同样有用，后续研究也发现存在冗余头、可裁剪头等现象。

## 13. 两种多头实现方式的对比

### 13.1 `MultiHeadAttentionWrapper`：最直观，但不够高效

这个版本本质上是：

- 直接创建多个 `CausalAttention` 子模块；
- 每个头各算各的；
- 最后 `torch.cat` 拼起来。

这个实现的优点是：

- 非常容易理解；
- 结构和"多个头并行"这个概念高度一致。

缺点是：

- 有较多 Python 层开销；
- 权重和计算更分散；
- 不利于框架底层做融合优化。

### 13.2 `MultiHeadAttention`：真实工程更接近这一种

这个版本只保留一套大的 `W_q / W_k / W_v`：

- 先把输出维度投到 `d_out`
- 再 reshape 成 `(batch, tokens, heads, head_dim)`
- 再 transpose 成 `(batch, heads, tokens, head_dim)`
- 在 head 维度并行完成注意力计算
- 最后拼回 `(batch, tokens, d_out)`

这更接近工业界的实现方式，因为：

- 张量布局更适合 GPU 并行；
- 更容易接入 fused kernel；
- 和底层库（PyTorch SDPA、FlashAttention、xFormers 等）的接口形式更一致。

### 为什么还要一个 `out_proj`

这个输出投影层 `W_O` 有两个作用：

- 把多头拼接后的信息重新混合；
- 让不同头之间发生线性组合，而不是简单并列摆放。

从研究角度看，`out_proj` 不一定是绝对必要的；但在主流 LLM 实现里，它仍然是标准配置。

## 14. 注意力实现里的张量形状，是最容易卡人的地方

写 attention 时，很多 bug 不是原理错误，而是 shape 错误。

### 单头常见形状

- 输入：`(T, d_in)`
- `Q/K/V`：`(T, d_out)`
- `scores = Q @ K.T`：`(T, T)`
- `weights @ V`：`(T, d_out)`

### 多头常见形状

- 输入：`(B, T, d_model)`
- 线性投影后：`(B, T, d_model)`
- reshape：`(B, T, H, D)`，其中 `D = d_model / H`
- transpose：`(B, H, T, D)`
- score：`(B, H, T, T)`
- 输出：`(B, H, T, D)`
- 合并 heads：`(B, T, d_model)`

### 这里有两个工程细节要记住

- `transpose` 后若要 `view`，通常需要先 `contiguous()`
- `d_out % num_heads == 0` 必须成立，否则头维度无法平均切分

这两个点在典型的 `MultiHeadAttention` 实现中都会体现得很明显。

## 15. 数学上优雅，但计算上昂贵：attention 的真正瓶颈是什么

经典 self-attention 的核心问题是：

\[
QK^T \in \mathbb{R}^{T \times T}
\]

一旦序列长度 `T` 变大，两个成本都会迅速上升：

- **计算量**：大致随 \(T^2\) 增长
- **显存占用**：注意力矩阵本身也是 \(T^2\)

这就是为什么注意力机制虽然表达能力强，但长上下文会很贵。

### 一个很容易忽略的事实

真正压垮系统的，往往不只是 FLOPs，而是：

> **显存带宽和中间张量读写。**

也就是说，很多时候 GPU 不是"算不动"，而是"搬数据太慢"。

这正是工业界后面发明 `FlashAttention` 的核心背景。

## 16. 工业界实践一：为什么要用 `scaled_dot_product_attention`

在教学实现里，我们一般手写：

```python
scores = q @ k.transpose(-2, -1)
scores = scores / math.sqrt(d_k)
scores = scores.masked_fill(mask, float('-inf'))
weights = torch.softmax(scores, dim=-1)
weights = dropout(weights)
out = weights @ v
```

这没问题，但工业界更常见的是使用框架提供的 fused 接口，例如 PyTorch 的：

```python
torch.nn.functional.scaled_dot_product_attention(...)
```

### 为什么

因为官方接口可以根据输入情况自动选择高效后端，例如：

- FlashAttention-2
- Memory-Efficient Attention
- PyTorch C++ math kernel

### 这意味着什么

你写的是同一个 attention 数学公式，但框架可能在底层替你做了：

- kernel fusion
- 更少的中间张量物化
- 更好的显存读写模式
- 更快的 CUDA 路径

### 需要注意的陷阱

PyTorch 官方文档特别提醒：

> `scaled_dot_product_attention` 会严格按照你传入的 `dropout_p` 执行 dropout，**不会自动因为 `eval()` 而关闭**。

所以工程里通常要显式写成：

```python
dropout_p = self.p if self.training else 0.0
```

这是教材实现和真实生产代码之间非常典型的差异。

---

## 17. 工业界实践二：FlashAttention 为什么快

FlashAttention 的核心不是"换了一个近似公式"，而是：

> **在数学等价的前提下，重写 attention 的内核执行方式，减少 HBM（显存）读写。**

### 它优化的不是哪一步公式，而是数据流

标准实现往往会：

- 显式构造较大的中间 attention 矩阵；
- 在显存和计算单元之间频繁搬运数据；
- 序列一长，IO 成本迅速爆炸。

FlashAttention 的思路则是：

- 分块（tiling）处理 Q/K/V；
- 尽量在片上高速存储中完成局部计算；
- 避免把完整 attention 矩阵长期物化在 HBM 中；
- 通过在线 softmax / 重计算等技巧控制数值正确性与内存占用。

### 所以它为什么重要

它说明一个非常关键的工程事实：

> **attention 的优化重点常常不是"改模型结构"，而是"改内核 IO 路径"。**

也正因为如此，在大模型训练和长上下文推理里，FlashAttention 这类实现优化往往能带来非常实在的收益。

---

## 18. 工业界实践三：训练和推理关注点完全不同

很多人学 attention 时默认"训练和推理就是同一套前向"，这在概念上没错，但在系统层面差别很大。

### 18.1 训练时更关注吞吐

训练通常是整段并行：

- 一次输入整段序列；
- 需要完整因果 mask；
- 更关心 batch 吞吐、显存占用、反向传播效率。

### 18.2 推理时更关注单 token 延迟

自回归生成是一个 token 一个 token 地往外吐：

- 先做一次 **prefill**：把 prompt 全部编码进去；
- 然后进入 **decode**：每步只新增一个 token。

这时新的瓶颈出现了：

- 之前的 `K/V` 会被反复使用；
- 如果每步都重新计算整个历史的 `K/V`，代价太大；
- 因此就引出了 **KV Cache**。

---

## 19. 工业界实践四：KV Cache 为什么能显著加速生成

对于 decoder-only 模型，在生成第 \(t\) 个 token 时：

- 新的 query 只和历史 key/value 交互；
- 历史 token 的 key/value 并没有变；
- 因此没必要每一步都重新算历史部分。

### KV Cache 的核心思想

把历史 token 的 `K` 和 `V` 缓存下来：

- 第一次 prefill 时全部算出；
- 后续 decode 每步只算新 token 的 `Q/K/V`；
- 再把新的 `K/V` 追加到缓存中。

### 它解决了什么

它不是把理论总复杂度神奇地降到线性，而是：

- **避免重复计算历史 K/V**
- 大幅降低逐步生成时的冗余计算
- 让单 token 推理延迟明显下降

### 新问题是什么

KV Cache 也不是免费的：

- 它会占用大量显存；
- 上下文越长，cache 越大；
- 很多推理系统最后卡的不是算力，而是 KV cache 内存。

所以在真实服务里，attention 优化经常演变为：

> **如何在"少算"和"少存"之间做平衡。**

---

## 20. 工业界实践五：为什么会出现 MQA / GQA

经典 Multi-Head Attention（MHA）中，每个 head 都有自己的 `Q/K/V`。

这在推理阶段有个问题：

- head 越多，`K/V cache` 越大；
- 大模型长上下文时，KV cache 成本很高。

于是工业界出现了两个很重要的折中方案：

### 20.1 MQA（Multi-Query Attention）

- 多个 query head
- 共享一组 key/value

优点：

- 显著减少 KV cache
- 提高推理吞吐

缺点：

- 头之间共享过多，表达能力可能受损

### 20.2 GQA（Grouped-Query Attention）

- query 头分组
- 每组共享一组 key/value

它位于 MHA 和 MQA 之间：

- 比 MHA 更省 cache
- 比 MQA 更保留质量

### 为什么现在很多模型偏爱 GQA

因为它在现实服务里很划算：

- 推理更友好
- 质量退化可控
- 对长上下文部署更实用

所以如果你只学教材里的标准 MHA，会理解原理；但如果你要理解今天的大模型服务系统，**GQA/MQA 是必须知道的工程演化方向。**

---

## 21. Attention 本身不编码顺序信息

单独的 self-attention 对输入集合是"内容交互友好"的，但对顺序本身并不天然敏感。

换句话说，如果没有位置编码：

- "猫咬狗"
- "狗咬猫"

从 token 集合角度可能很像，但语义完全不同。

所以真实 Transformer 里，attention 机制几乎总要和位置机制一起工作：

- 绝对位置编码
- 相对位置编码
- RoPE（旋转位置编码）
- ALiBi 等

本文主要聚焦 attention 核心本体，没有展开位置编码，但在工业实践里，这两者是一起考虑的。

> **没有位置机制的 attention，不足以成为语言模型的完整序列建模器。**

---

## 22. 长上下文时代，为什么"标准 attention"还不够

即便用了 FlashAttention，标准 dense attention 的本质仍是全连接两两交互，因此长上下文仍然昂贵。

这催生了很多工程和结构上的折中：

- 滑动窗口 attention
- 分块 attention
- 稀疏 attention
- 压缩记忆
- 检索增强（RAG）
- 线性 attention / state-space alternatives

### 工程现实

在今天的大模型工业实践里：

- **dense attention 仍然是主力基线**
- 真正广泛落地的优化，往往先是 **内核优化 + cache 优化 + head 共享优化**
- 结构级替代方案虽然很多，但不是所有方案都能稳定替代标准 attention

所以从工程角度看，最值得先掌握的仍然是：

- 标准 attention 的数学本体
- causal mask
- multi-head
- FlashAttention / SDPA
- KV cache
- GQA/MQA

## 23. 从注意力机制中学到的，不只是代码

如果把这章压缩成几条真正有迁移价值的认知，我认为是下面这些：

### 23.1 注意力不是魔法，就是"相似度检索 + 加权聚合"

先算分数，再归一化，再聚合 value。

### 23.2 Q/K/V 分离是能力跃迁的关键

它把"怎么找"和"找到后拿什么"拆开了。

### 23.3 缩放不是装饰，是训练稳定性的关键

没有 \(\sqrt{d_k}\) 缩放，softmax 很容易过饱和。

### 23.4 causal mask 的位置不能错

必须在 softmax 之前施加。

### 23.5 multi-head 的价值在于多视角检索

不是简单堆参数，而是让模型同时学习多种关系模式。

### 23.6 教学实现和工业实现之间隔着"内核与系统"

教材会教你公式和 shape，工业界会逼你面对：

- 显存带宽
- fused kernel
- cache 管理
- 推理延迟
- 多卡吞吐

## 24. 一个更接近工程直觉的最小实现框架

下面不是对教材代码的复写，而是一个"你脑中应该留下来的 attention 框架"：

```python
# x: (B, T, d_model)
q = Wq(x)
k = Wk(x)
v = Wv(x)

q = split_heads(q)  # (B, H, T, D)
k = split_heads(k)
v = split_heads(v)

scores = q @ k.transpose(-2, -1) / sqrt(D)   # (B, H, T, T)
scores = scores.masked_fill(causal_mask, -inf)
weights = softmax(scores, dim=-1)
weights = dropout(weights)                    # training only
out = weights @ v                             # (B, H, T, D)
out = merge_heads(out)                        # (B, T, d_model)
out = Wo(out)
```

如果你能把这段流程和下面这些问题一一对应起来，就算真正理解了：

- 为什么要 `Wq/Wk/Wv`
- 为什么要 `/ sqrt(D)`
- 为什么 mask 在 softmax 前
- 为什么多头需要 split / merge
- 为什么有 `Wo`
- 为什么工业界会用 SDPA / FlashAttention 来替代手写矩阵步骤

## 25. 常见误区总结

### 误区 1：attention 就是在"找相似词"

不完全对。它学到的是"对当前任务有用的关系"，不只是表面相似。

### 误区 2：多头一定每个都很重要

不一定。真实模型里常有冗余头。

### 误区 3：有了 causal mask 就万无一失

还不够。推理时还要考虑 cache、位置编码、padding、batch 打包等细节。

### 误区 4：FlashAttention 是一种新的 attention 公式

不是。它主要是**更高效的实现方式**。

### 误区 5：KV cache 让生成复杂度彻底变线性

不准确。它减少了历史 K/V 的重复计算，但长上下文下缓存本身也会成为系统瓶颈。

## 26. 最后的整体总结

如果只用一句话概括注意力机制：

> **注意力机制让每个 token 都能基于当前需求，从整段序列中动态检索有用信息，并把这些信息融合成新的表示。**

如果再加一句工程视角：

> **现代大模型的 attention，数学核心仍然是 scaled dot-product attention，但真正决定训练吞吐和推理延迟的，往往是 mask 设计、张量布局、内核实现、KV cache，以及 MHA 到 GQA/MQA 的工程折中。**

所以，这一章最值得掌握的不是"记住一个公式"，而是建立下面这条完整链路：

- 从相似度加权平均出发理解 attention
- 理解 Q/K/V 为什么必须分离
- 理解缩放、softmax、mask 的数学原因
- 理解 single-head 到 multi-head 的表达力提升
- 理解为什么工业界还要做 FlashAttention、KV cache、GQA/MQA 这些改造

建立这条完整链路之后，理解任何 Transformer 变体都会容易得多。
