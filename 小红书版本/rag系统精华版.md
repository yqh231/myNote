# RAG系统全解：让AI不再胡说八道的核心技术

> 本文提炼自 RAG 系统学习笔记（基于 HelloAgents 项目），适合快速了解 RAG 的核心原理和工程设计。

---

## 一、RAG是什么？为什么需要它？

RAG（Retrieval-Augmented Generation，检索增强生成），核心思路就是**"先检索、再生成"**——让LLM回答问题之前，先从外部知识库中检索相关资料注入Prompt，弥补模型知识的静态性和有限性。

简单说：不让AI纯靠记忆回答，而是先"查资料"再回答。

---

## 二、RAG的三代进化

### 朴素RAG
最早的形态，流程是"检索-读取"模式。用TF-IDF、BM25等关键词匹配算法检索文档，把检索结果原文拼进Prompt交给LLM。

问题很明显：只能做字面匹配，搜"ML"找不到"机器学习"，检索质量差时反而给LLM引入噪声。

### 高级RAG
引入了**向量语义检索**——把文本映射为高维向量，通过余弦相似度计算语义距离，能理解同义词和近义表达。同时在流程上加入了查询重写、文档分块、重排序等优化。

检索质量大幅提升，但各模块之间仍然是硬编码的线性流程，灵活性不足。

### 模块化RAG（当前主流）
将系统拆分为可插拔、可组合的独立模块。检索层可以自由组合向量检索、关键词检索、图谱检索等策略；生成层可以引入思维链推理、自我反思、上下文压缩等技术。

核心优势：**每一层可以独立替换和优化**——换嵌入模型不影响业务逻辑，换向量数据库不影响检索策略。

三代对比速览：

| 维度 | 朴素RAG | 高级RAG | 模块化RAG |
|------|--------|--------|----------|
| 检索方式 | TF-IDF / BM25 | Dense Embedding | 混合检索 + MQE + HyDE |
| 文档处理 | 无/简单切分 | 固定窗口分块 | 结构感知智能分块 |
| 可扩展性 | 差 | 中等 | 各模块可插拔替换 |

## 三、检索策略：如何从知识库里找到对的东西

检索是RAG的核心环节，直接决定最终生成质量的上限。

### 三种基础检索方式

**关键词检索**：按词频匹配，速度快、实现简单，适合精确术语查找，但无法处理同义词。

**向量语义检索**：把查询和文档都转为嵌入向量，通过余弦相似度计算语义距离。能理解"机器学习"和"ML"是同一概念，是RAG的主力检索方式。

**图谱检索**：通过知识图谱做多跳推理。比如查"张三的同事负责什么项目"，需要先找到张三，沿"同事"关系找到人，再沿"负责"关系找到项目。纯向量检索做不了这种关联推理。

### 混合检索：单一方式各有盲区

实际系统中，不同类型的数据适合不同的检索组合。以HelloAgents为例：

| 记忆类型 | 混合方式 | 评分公式 |
|---------|---------|---------|
| 工作记忆 | TF-IDF + 关键词 | (向量x0.7 + 关键词x0.3) x 时间衰减 x 重要性 |
| 情景记忆 | 结构化过滤 + 向量 | (向量x0.8 + 时间近因性x0.2) x 重要性 |
| 语义记忆 | 向量 + 图谱 | (向量x0.7 + 图谱x0.3) x 重要性 |

设计逻辑：情景记忆（具体事件）强调时间近因性；语义记忆（抽象知识）引入图谱发现隐含关系；工作记忆数据量小，TF-IDF足以应对。

## 四、两个高级检索神技

### MQE（多查询扩展）：一个问题，多种问法

**要解决的问题**：用户的表述和知识库中的文档用词经常不一致。用户问"如何学Python"，文档标题是"Python入门指南"，语义一致但字面不同，这就是**词汇鸿沟**。

**核心原理**：既然一种表述不够，那就用多种表述。让LLM把原始查询改写为多个语义等价但用词不同的查询，分别检索后合并去重。

**工作流程**：
1. 原始查询"如何学习Python"送给LLM
2. LLM生成扩展查询："Python入门教程"、"Python学习方法"
3. 三条查询并行检索，各自拿回一批候选文档
4. 同一文档出现多次只保留最高分，按分数排序返回top-k

**关键设计**：
- 默认生成2条扩展，加上原始查询共3条，平衡召回率和开销
- Prompt要求"简短、避免标点"，让扩展查询接近检索query风格而非完整句子
- LLM调用失败时自动回退到原始查询，保证流程不中断
- 效果：召回率提升30%-50%，对模糊查询和专业术语效果显著

**局限**：MQE本质是查询侧的同义改写，能解决"用词不同"但解决不了"句式不同"——用户问的是疑问句，文档写的是陈述句，两者在向量空间中的分布本身就不同。

### HyDE（假设文档嵌入）：用答案找答案

**要解决的问题**：问题和答案在向量空间中往往不相近。"什么是自注意力机制？"是疑问句，"自注意力机制通过计算序列中各位置间的关联权重..."是陈述句，句式结构和用词模式都不同，导致嵌入向量存在系统性偏差。这就是**查询-文档语义鸿沟**。

**核心原理**：先让LLM生成一个"假设性答案"，再用假答案的向量去检索真实文档。假答案内容不一定准确，但它的**表述形式**（陈述句）和**关键术语**与真实文档高度相似，在向量空间中更容易匹配到正确文档。

**工作流程**：
1. 用户问"Transformer的自注意力机制是什么？"
2. LLM生成假设答案："自注意力机制是Transformer架构的核心组件，通过计算输入序列中每个位置与所有其他位置之间的注意力权重..."
3. 对假设答案（而非原始问题）做向量编码
4. 用该向量在知识库中检索真实文档

**为什么假答案不准确也能工作**：假设LLM把"softmax归一化"写成了"L2归一化"，但假答案中仍然包含"自注意力"、"Transformer"、"Query/Key/Value"、"注意力权重"等领域核心术语。HyDE利用的不是假答案的**事实准确性**，而是它的**语言形式和术语分布**。

**特别有效的场景**：
- 专业领域查询：用户不知道精确术语，LLM能在假设答案中补充
- 跨语言检索：中文提问，假设答案包含英文术语，匹配到英文文档
- 复杂概念查询：假设答案展开后包含大量关联概念

**局限**：依赖LLM生成质量；对简单事实查询开销过大；每次多一次LLM调用增加延迟。

### MQE + HyDE 联合使用

两者互不冲突，同时启用覆盖面最广：MQE从多个角度扩展查询用词，HyDE从答案视角桥接语义鸿沟。

| 维度 | MQE | HyDE |
|------|-----|------|
| 解决的问题 | 用词差异（同义表达） | 句式差异（问句 vs 陈述句） |
| 改变的对象 | 查询的措辞 | 查询的形式 |
| LLM输出量 | 短（几个短句） | 长（一个段落） |
| 适合的查询类型 | 模糊查询、口语化表述 | 专业查询、概念性问题 |

策略选择建议：日常查询用基础+MQE；专业领域用MQE+HyDE；延迟敏感只用基础检索；知识库小也只用基础检索。

## 五、三个数据库各司其职

RAG系统不只需要一个数据库，不同类型的查询需要不同的存储引擎。

### Qdrant（向量数据库）——找语义相似的
所有语义相似度检索的场景都由它承担。将文本转为嵌入向量存入，检索时通过HNSW索引快速找到最相似的top-k个向量。支持Payload过滤，检索时可同时做语义匹配和条件筛选。百万级数据仍保持毫秒级响应。

### Neo4j（图数据库）——找关系关联的
专门服务于知识图谱，存储实体之间的关系网络。向量检索擅长"找相似的"，但无法处理"A和B是什么关系"这类结构化推理。比如查"和张三相关的技术领域"，图遍历可以沿 `张三 -> 专注于 -> 机器学习` 找到答案，向量检索可能找不到这种间接关联。

### SQLite（关系型数据库）——管结构化记录的
存储记忆的完整元信息（ID、内容、类型、重要性、时间戳等），支持精确条件查询和统计分析。在检索流程中做第一轮结构化预过滤（时间范围、重要性等），缩小候选集后再交给Qdrant做向量检索。零配置零运维，单文件存储。

### 三库协作模式

```
用户查询 -> MemoryManager统一调度
  - 工作记忆: 纯内存（无外部数据库）
  - 情景记忆: SQLite(结构化过滤) + Qdrant(语义检索)
  - 语义记忆: Qdrant(语义检索) + Neo4j(关系推理)
  - 感知记忆: Qdrant(多模态向量检索)
  - RAG检索:  Qdrant(文档向量检索)
```

每个数据库只做自己最擅长的事，且任一组件都可以独立替换，不影响整体架构。

## 六、排序算法：检索完了，谁排前面？

检索只是找到候选文档，**排序决定了用户最终看到什么**。不同类型的数据需要不同的排序策略。

### 工作记忆：指数时间衰减
工作记忆是"临时便签"，越新越重要。采用指数衰减函数，每6小时衰减一个周期，几小时前的信息就被显著降权。这比线性衰减更符合人类短期记忆的遗忘规律。

基础分 = TF-IDF向量相似度 x 0.7 + 关键词匹配 x 0.3，再乘以时间衰减和重要性权重。

### 情景记忆：倒数时间衰减
存储具体事件，时间跨度从几天到几个月，衰减需要更平缓。采用倒数衰减 `recency = 1/(1+天数)`：当天事件满分，1天前0.5，一周前约0.14，一个月前约0.03。尾部比指数衰减更长，几周前的事件不会被完全忽略。

基础分 = 向量相似度 x 0.8 + 时间近因性 x 0.2。

### 语义记忆：向量+图谱双通道融合（最复杂）
知识没有时效性（"Python是解释型语言"不会过期），所以不引入时间衰减。

同时使用向量通道和图谱通道：向量通道返回余弦相似度；图谱通道返回实体匹配度(x0.6)、实体密度(x0.2)、关系密度(x0.2)的复合评分。双通道融合：向量分 x 0.7 + 图谱分 x 0.3。

特别的是，语义记忆用Softmax对最终分数做概率归一化，提供"这条结果有多大置信度"的信息——当所有候选分数都很低时，概率趋于均匀分布，暗示"没有明确的最佳答案"。

### 统一的重要性权重
四种记忆共享同一套重要性权重：`weight = 0.8 + importance x 0.4`，映射到[0.8, 1.2]区间，最多让分数浮动正负20%，避免重要性过度干扰相似度排序。

### 和业内方法的对比

HelloAgents用的是**多因子加权融合**，业内还有几种主流方法：

- **RRF（倒数排名融合）**：不看绝对分数只看排名位置，当不同检索通道的分数量纲差异大时更稳健
- **交叉编码器重排序**：用专门的模型对初筛结果做二次精排，精度更高但有额外模型开销
- **Learning to Rank**：用机器学习从数据中自动学习排序权重，需要标注数据
- **多阶段排序**：生产级系统常用漏斗式 召回(~1000条) -> 粗排(~100条) -> 精排(~20条) -> 重排(~5条)

## 七、总结

1. **RAG三代进化**：朴素(关键词) -> 高级(语义检索) -> 模块化(可插拔组件)
2. **检索三个层次**：基础检索 -> 混合检索 -> 高级策略(MQE+HyDE)
3. **三库分工**：Qdrant找相似的，Neo4j找相关的，SQLite管记录的
4. **排序核心设计**：统一重要性权重 + 差异化时间衰减 + 语义记忆双通道融合

一句话：**检索决定上限，排序决定体验，模块化决定可维护性。**
