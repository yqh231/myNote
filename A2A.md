# 智能体通信协议: MCP、A2A 与 ANP

> 参考资料: 《Hello Agents》第十章 智能体通信协议
>
> 本文梳理当前智能体生态中三种主流通信协议的设计思想、运作机制与适用场景，并结合业界实践进行分析。

## 1 为什么需要智能体通信协议

单体智能体在具备推理、工具调用和记忆能力后，仍面临三个根本性瓶颈:

1. **工具集成碎片化**: 每接入一个新的外部服务(GitHub API、数据库、文件系统等)，都需要手写一套适配器代码，处理 HTTP 请求、认证、错误处理等，且不同开发者编写的工具无法互通。
2. **能力边界固化**: 智能体的能力被限制在预先定义的工具集内，无法在运行时动态发现和使用新服务。
3. **多智能体协作缺失**: 当任务需要多个专业智能体(如研究员、撰写员、编辑)协作时，缺乏标准化的通信手段，只能依赖硬编码的编排逻辑。

通信协议的核心价值在于提供**标准化接口**，让不同的智能体、工具、服务能够以统一的方式连接和交互。这就像 TCP/IP 让异构网络设备能够互通一样，智能体通信协议让 AI 生态中的各种组件能够"即插即用"。

## 2 三种协议总览

| 维度 | MCP | A2A | ANP |
|------|-----|-----|-----|
| 提出者 | Anthropic | Google | 开源社区 |
| 核心定位 | 智能体与工具的桥梁 | 智能体间的点对点协作 | 大规模智能体网络基础设施 |
| 解决的核心问题 | 如何标准化访问外部工具/资源 | 如何让智能体相互对话与协作 | 如何在大规模网络中发现和连接智能体 |
| 设计哲学 | 上下文共享 | 对等通信 | 去中心化服务发现 |
| 通信模式 | Client-Server | Peer-to-Peer | 服务注册与发现 |
| 生态成熟度 | 相对成熟，社区活跃 | 早期阶段，Google 背书 | 概念性框架，社区探索中 |

一个简单的类比: MCP 是"USB-C 接口"，统一了设备(工具)的连接方式；A2A 是"电话通信协议"，让人(智能体)之间能够对话；ANP 是"电话号码簿+交换机"，让你在茫茫网络中找到需要的人。

## 3 MCP: 标准化工具访问

### 3.1 设计思想

MCP (Model Context Protocol) 由 Anthropic 提出，其核心目标是**统一智能体与外部工具/资源的交互方式**。

传统做法下，智能体需要为每个外部服务编写专门的 Tool 类，不同 LLM 平台的 Function Calling 实现差异巨大(OpenAI 用 `parameters`，Claude 用 `input_schema`)，切换模型几乎要重写一遍工具层代码。MCP 通过定义一套标准协议，将工具的提供方(Server)和消费方(Client)解耦，让任何支持 MCP 的模型都能无缝访问相同的工具集。

### 3.2 三层架构

MCP 采用 Host-Client-Server 三层架构:

- **Host(宿主层)**: 用户直接交互的界面(如 Claude Desktop、IDE 插件)，负责管理对话流程。
- **Client(客户端层)**: 协议通信的中间层，负责与 MCP Server 建立连接、发送请求和接收响应。
- **Server(服务器层)**: 实际的功能提供方，封装具体的工具逻辑(文件操作、API 调用、数据库查询等)。

这种**关注点分离**的设计让开发者只需专注于编写 MCP Server，无需关心 Host 和 Client 的实现细节。

### 3.3 三大核心能力

| 能力 | 性质 | 说明 |
|------|------|------|
| Tools(工具) | 主动的 | 执行操作，如调用 API、写入文件 |
| Resources(资源) | 被动的 | 提供数据，如文件内容、数据库记录 |
| Prompts(提示模板) | 指导性的 | 提供预定义的交互模板 |

### 3.4 工作流程

完整的工具调用链路:

1. **工具发现**: MCP Client 连接 Server 后调用 `list_tools()` 获取所有可用工具的名称、描述、参数定义。
2. **上下文构建**: Client 将工具列表转换为 LLM 能理解的格式，注入到系统提示词中。
3. **模型推理**: LLM 分析用户问题和可用工具，自主决定是否调用工具以及调用哪个工具。
4. **工具执行**: Client 通过 MCP Server 执行被选中的工具，获取结果。
5. **结果整合**: 工具返回值送回 LLM，LLM 结合结果生成最终回答。

整个过程是自动化的，LLM 根据工具描述的质量来做决策，这也意味着**编写清晰准确的工具描述至关重要**。

### 3.5 传输方式

MCP 的一个重要特性是**传输层无关性**，支持多种传输方式:

| 传输方式 | 适用场景 | 特点 |
|----------|----------|------|
| Memory | 单元测试、原型开发 | 内存传输，无网络开销 |
| Stdio | 本地开发、调试 | 通过标准输入输出与本地进程通信 |
| HTTP | 生产环境、远程服务 | 标准 HTTP 请求/响应 |
| SSE | 实时通信、流式处理 | 服务端推送事件，适合长连接 |
| StreamableHTTP | 双向流式通信 | 结合 HTTP 与流式能力 |

### 3.6 MCP 与 Function Calling 的关系

这是一个常见的困惑点。二者并非竞争关系，而是不同层次的能力:

- **Function Calling** 是 LLM 的内在能力，体现模型的智能——知道何时调用函数、如何生成参数。相当于你学会了"如何打电话"。
- **MCP** 是基础设施协议，在工程层面解决工具与模型的连接问题，提供标准化的描述和调用方式。相当于全球统一的"电话通信标准"。

Function Calling 因模型而异(OpenAI、Claude、Gemini 各有格式)，而 MCP 提供统一抽象层，一套工具可以被任何支持 MCP 的模型复用。

### 3.7 业界实践与生态

MCP 目前拥有三种协议中最成熟的生态:

- **Anthropic 官方维护**的 MCP Server(文件系统、GitHub、PostgreSQL、Google Drive 等)质量最高。
- **社区资源**如 Awesome MCP Servers (https://github.com/punkpeye/awesome-mcp-servers)、MCP Servers 官方目录 (https://mcpservers.org/) 提供了大量第三方实现。
- 主流 IDE 和 AI 工具(Claude Desktop、Cursor、Continue 等)已原生支持 MCP。

**实际应用场景举例**:
- **自动化测试**: 通过 Playwright MCP Server，智能体可以自动打开浏览器、填写表单、截图验证。
- **项目管理**: 结合 Jira + GitHub MCP Server，实现 Issue 到任务的自动同步。
- **知识管理**: 结合 Perplexity + Obsidian MCP Server，智能体搜索最新资讯并自动整理成结构化笔记。

**需要注意的问题**: 社区 MCP Server 的质量参差不齐，时效性取决于维护者，生产环境建议优先选择大公司背书的官方实现。

## 4 A2A: 智能体间的对等协作

### 4.1 设计动机

MCP 解决了智能体"如何使用工具"的问题，但当一个复杂任务需要多个专业智能体协作时(比如"研究员"搜集资料、"撰写员"写文章、"编辑"审校优化)，它们之间需要能够通信、委托任务、协商能力和同步状态。

传统的**中央协调器(星型拓扑)**方案存在三大问题:
- **单点故障**: 协调器一旦宕机，整个系统瘫痪。
- **性能瓶颈**: 所有通信必须经过中心节点，限制并发吞吐。
- **扩展困难**: 增删智能体需要修改中心编排逻辑。

A2A (Agent-to-Agent Protocol) 由 Google 提出，采用**点对点(P2P)架构(网状拓扑)**，允许智能体直接通信，从根本上解决上述问题。

### 4.2 核心概念

A2A 的核心抽象是**Task(任务)**和**Artifact(工件)**:

| 概念 | 说明 |
|------|------|
| Task(任务) | 智能体间协作的基本单元，包含完整的生命周期管理 |
| Artifact(工件) | 任务执行的产物，如生成的文档、分析报告、代码等 |
| Skill(技能) | 智能体对外暴露的能力，如"research"、"write"、"edit" |
| Agent Card | 智能体的"名片"，描述其名称、能力、接口等元信息 |

这是 A2A 与 MCP 的关键区别: MCP 围绕"工具调用"设计，关注的是单次请求-响应；A2A 围绕"任务"设计，一个任务可能经历创建、协商、执行、完成等多个阶段。

### 4.3 任务生命周期

A2A 为任务定义了标准化的状态机:

```
创建 -> 协商 -> 代理 -> 执行中 -> 完成/失败
```

这套机制让智能体之间可以进行:
- **任务协商**: Agent B 收到 Agent A 的任务请求后，可以接受、拒绝或提出反提案(如"时间太紧，建议延长到7天")。
- **进度跟踪**: 请求方可以随时查询任务状态。
- **异常处理**: 任务失败时可以自动触发重试或降级策略。

### 4.4 请求生命周期

A2A 请求遵循四个主要步骤:

1. **Agent 发现**: 客户端找到目标智能体的服务端点(通过 Agent Card)。
2. **身份验证**: 确认双方身份的合法性。
3. **发送消息 API**: 同步的请求-响应模式。
4. **发送消息流 API**: 异步的流式通信模式，适用于长时间任务。

### 4.5 运作方式

A2A 的典型使用模式:

**服务端**: 每个智能体作为 A2A Server 运行，通过装饰器注册技能:

```python
researcher = A2AServer(name="researcher", description="研究员")

@researcher.skill("research")
def handle_research(text: str) -> str:
    # 实际的研究逻辑
    return research_result

researcher.run(host="localhost", port=5000)
```

**客户端**: 其他智能体通过 A2A Client 发起调用:

```python
client = A2AClient("http://localhost:5000")
response = client.execute_skill("research", "AI在医疗领域的应用")
```

**多智能体协作网络**: 多个 Agent 组成流水线:

```
研究员(port:5000) -> 撰写员(port:5001) -> 编辑(port:5002)
```

每个环节的输出作为下一个环节的输入，实现"研究 -> 撰写 -> 审校"的完整内容生产流程。

### 4.6 业界实践分析

A2A 目前处于发展早期，Google 虽然给出了协议规范和 Sample Code，但成熟的生产级实现还比较少。实际落地时需要注意:

**优势**:
- 对等架构避免了中心化瓶颈，理论上具有更好的扩展性。
- 任务生命周期管理提供了比简单 RPC 更丰富的协作语义。
- Google 背书带来一定的行业影响力。

**当前局限**:
- 生态尚不成熟，现有实现多为示例代码，缺乏生产级 SDK。
- 服务发现机制不够完善，Agent Card 的分发和更新依赖外部机制。
- 与 MCP 的定位有一定交叉(都涉及"调用远程能力")，开发者在选择时可能产生困惑。

**与 MCP 的协同**: 在实际架构中，MCP 和 A2A 并不冲突。一个典型的设计是: 每个智能体内部通过 MCP 访问工具(文件系统、数据库等)，智能体之间通过 A2A 进行协作通信。MCP 解决"纵向"的工具集成，A2A 解决"横向"的智能体协作。

## 5 ANP: 大规模智能体网络

### 5.1 设计目标

当智能体的数量从几个扩展到成百上千，一系列新的挑战浮现:

- **服务发现**: 新任务到达时，如何快速找到能处理该任务的智能体?
- **智能路由**: 多个智能体都能处理同一任务时，如何根据负载、成本、性能等因素选择最优?
- **动态扩展**: 新加入网络的智能体如何被其他成员发现和调用?

ANP (Agent Network Protocol) 是一个由开源社区维护的概念性协议框架，专注于构建**大规模、去中心化的智能体网络基础设施**。

### 5.2 核心概念

| 概念 | 说明 |
|------|------|
| Discovery(服务发现) | 注册中心，维护所有可用智能体的索引 |
| Service(服务) | 每个智能体注册的元信息，包括 ID、类型、能力、端点、负载等 |
| Network(网络) | 智能体之间的拓扑结构，包括节点和连接关系 |
| DID(去中心化身份) | 基于去中心化标识符的身份验证机制 |

### 5.3 运作流程

ANP 的整体工作流程:

1. **服务注册**: 智能体启动时，向发现服务注册自己的元信息(ID、类型、能力列表、端点地址、负载状态等)。
2. **服务发现与匹配**: 需要某种能力时，通过发现服务基于语义或功能描述进行查询。发现服务通过预先爬取各智能体的标准端点(`.well-known/agent-descriptions`)来建立索引。
3. **身份验证**: 基于 DID(去中心化身份标识符)的双向验证。智能体 A 用私钥签名请求，智能体 B 通过解析 DID 获取公钥验证签名的真实性。
4. **智能路由与负载均衡**: 当多个智能体都满足需求时，根据负载、CPU 核心数、内存、GPU 等元数据进行最优选择。
5. **服务执行**: 验证通过后，双方依据预定义的标准接口进行数据交换或服务调用。

### 5.4 实际应用模式

**服务注册与发现**:

```python
discovery = ANPDiscovery()

# 注册多个同类型服务
register_service(discovery, service_id="nlp_agent_1",
    service_type="nlp", capabilities=["text_analysis", "ner"],
    metadata={"load": 0.3, "price": 0.01})

register_service(discovery, service_id="nlp_agent_2",
    service_type="nlp", capabilities=["text_analysis", "translation"],
    metadata={"load": 0.7, "price": 0.02})

# 按类型查找 + 负载均衡
services = discover_service(discovery, service_type="nlp")
best = min(services, key=lambda s: s.metadata.get("load", 1.0))
```

**智能任务调度**: 结合 LLM 的推理能力，让智能体自主选择最优节点:

```python
# 注册10个计算节点(含 CPU/内存/GPU 等元数据)
# 智能调度器分析任务需求，自动匹配最合适的节点
assign_task("训练深度学习模型，需要 GPU 支持")  # -> 选择有 GPU 的低负载节点
assign_task("处理大量文本数据，需要高内存")        # -> 选择 64GB 内存节点
```

### 5.5 业界实践分析

ANP 目前是三种协议中成熟度最低的:

**潜在价值**:
- 填补了智能体生态中"服务发现与路由"这一空白，MCP 和 A2A 都没有内建完整的服务发现机制。
- DID 身份验证机制为去中心化信任提供了基础，适合开放网络环境。
- 负载均衡、智能路由等能力对大规模部署至关重要。

**当前状态**:
- 仍处于概念验证阶段，官方实现(AgentConnect)还在早期开发中。
- 缺乏广泛的行业采用和实战验证。
- 与 Kubernetes 等成熟的服务编排框架相比，目前没有明显的差异化优势。

**务实的建议**: 对于大规模智能体部署的服务发现和路由需求，短期内可以借助成熟的微服务基础设施(如 Consul、Etcd、Kubernetes Service)来实现类似能力，ANP 更适合作为长期的概念参考和技术储备。

## 6 三种协议的协同与选择

### 6.1 分层架构

在一个完整的智能体系统中，三种协议可以在不同层次协同工作:

```
应用层:  用户请求 -> 协调智能体
          |
协作层:  A2A (智能体间通信与任务协作)
          |
工具层:  MCP (每个智能体访问各自的工具和资源)
          |
网络层:  ANP (服务发现、路由、负载均衡)
```

### 6.2 选择指南

| 场景 | 推荐协议 | 理由 |
|------|----------|------|
| 智能体需要访问外部服务(文件、数据库、API) | MCP | 生态成熟，即插即用 |
| 多个智能体需要相互协作完成任务 | A2A | 专为智能体间对等通信设计 |
| 构建大规模智能体生态系统 | ANP | 提供服务发现和网络管理能力 |
| 快速原型开发 | MCP | 社区资源丰富，上手成本低 |
| 生产环境多智能体编排 | MCP + A2A | MCP 管工具，A2A 管协作 |

### 6.3 务实建议

当前协议生态仍处于发展早期，技术选型需要考虑实际成熟度:

1. **优先采用 MCP**: 它拥有最成熟的生态、最丰富的社区资源和最广泛的工具支持。对于大多数项目，MCP 就能满足核心需求。
2. **按需引入 A2A**: 当确实需要多智能体协作且简单的函数调用无法满足时，再考虑 A2A。注意目前 A2A 的实现多为示例级别，生产使用需要额外的工程投入。
3. **关注 ANP 演进**: ANP 的去中心化服务发现理念有价值，但短期内可以用成熟的微服务基础设施替代。建议作为技术雷达中的"评估"项持续关注。
4. **不要过度设计**: 如果你的系统只有 2-3 个智能体，简单的函数调用或 HTTP API 可能比引入完整的协议栈更高效。协议的价值在规模化场景下才能充分体现。
