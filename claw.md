## 一句话理解 OpenClaw

OpenClaw 是一个跑在你自己机器上的 AI 助手中枢。你在 Telegram 给它发一句话，它帮你干活，然后把结果回到 Telegram。换成 Discord、Slack、WhatsApp 也一样，它不挑通道。

它的核心定位是 **"The AI that actually does things"** -- 不只是聊天，而是能操作文件、执行命令、搜网页、甚至遥控你的手机。

项目用 TypeScript 写，约 100 万行代码，内置 80 多个插件、50 多个技能、支持 20 多个消息平台和 30 多个 LLM 模型。

## 它到底在干什么：一条消息的旅程

理解 OpenClaw 最好的方式，就是跟着一条消息从头走到尾。

假设你在 Telegram 发了一句"帮我查下北京天气"。

**第一站：通道适配。** Telegram 把这条消息推给 OpenClaw 的 Telegram 插件。Telegram 的消息格式是它自己那一套 JSON，插件的工作就是把它翻译成系统内部的统一格式 -- 谁发的、从哪个平台来的、内容是什么。翻译完之后，对系统内部来说，这条消息跟从 Discord 或 Slack 来的没有任何区别。

**第二站：路由。** 系统拿到这条标准化的消息后，第一个问题是"这条消息该交给哪个 Agent 处理"。OpenClaw 支持配置多个 Agent（比如一个负责日常聊天，一个负责技术支持），路由模块会根据消息来源（哪个聊天窗口、哪个群、哪个用户）去匹配预设的绑定规则，找到对应的 Agent。同时它会生成一个 Session Key，用来唯一标识这段对话。

**第三站：会话加载。** 根据 Session Key，系统从磁盘上找到这段对话的历史记录。如果是新对话就创建一个空的。会话里存着之前所有的来回消息，这样 AI 才知道上下文。

**第四站：排队。** 这里有一个关键设计：同一个会话的消息严格排队，一条处理完才处理下一条。这是为了防止你连发两条消息时，AI 同时处理两条导致上下文混乱。但不同会话之间是完全并行的，你和你朋友同时发消息互不影响。

**第五站：上下文组装。** 这是交给 AI 之前的准备工作。系统把 System Prompt（告诉 AI 它是谁、该怎么做）、对话历史、可用的工具列表（比如执行命令、搜索网页）打包在一起，组成一个完整的请求。同时它会确定用哪个模型 -- 比如你给这个 Agent 配的是 Claude，那就用 Claude；如果 Claude 挂了，会自动切到备用模型。

**第六站：AI 推理。** 打包好的请求发给 LLM Provider（比如 Anthropic 的 API）。AI 开始思考，返回的内容是流式的，一个字一个字往回吐。系统会实时把这些字推送给连着的客户端，所以你能看到 AI "正在输入"的效果。

**第七站：工具调用循环。** 这是最有意思的部分。AI 看到"查天气"这个需求，它不会瞎编，而是决定调用 web_search 工具去搜一下。系统收到这个工具调用请求后，先做安全检查（这个工具允许调用吗？输入合法吗？需要放沙箱里执行吗？），然后真正执行，拿到结果后再送回给 AI。AI 看到搜索结果，可能觉得够了，直接组织语言回复；也可能觉得还需要进一步操作，再调用另一个工具。这个"AI 思考 -> 调工具 -> 拿结果 -> 再思考"的循环会持续下去，直到 AI 认为可以直接回答了。

**第八站：回复发送。** AI 最终的回复是 Markdown 格式的纯文本。但 Telegram 需要 HTML 格式，Discord 用自己的 Markdown 方言，Slack 用 mrkdwn。通道插件负责把统一格式翻译回各平台的特定格式，然后调用平台 API 发出去。

**第九站：持久化。** 最后，系统把这轮对话（用户消息 + AI 回复 + 工具调用记录）追加到 Session 文件里，存到磁盘。下次对话就能接着之前的上下文继续。写文件用的是"先写临时文件再原子改名"的方式，防止写到一半断电导致数据损坏。

整个过程中，系统还会在 Telegram 里用 emoji 反应来告诉你当前状态：收到消息显示眼睛，思考中显示思考脸，调工具时显示火焰，完成了显示大拇指。

## 核心设计理念

### 一个网关管所有

OpenClaw 的架构核心是一个 Gateway 进程。所有的消息通道、AI 模型、工具执行、配置管理都在这一个进程里完成。它没有用微服务，因为个人 AI 助手不需要水平扩展，一台机器跑一个进程就够了。

这带来的好处是简单：所有内部通信都是函数调用，没有网络延迟；所有状态都在一个地方，不需要分布式一致性；部署就是启动一个进程，不需要编排容器集群。

Gateway 对外只开一个端口（默认 18789），用 WebSocket 通信。CLI、Web UI、手机 App 都通过这个端口连进来。协议很简单，就三种消息：请求、响应、事件。请求是客户端主动调 Gateway 的方法，响应是 Gateway 回的结果，事件是 Gateway 主动推的通知（比如 AI 正在输出的流式内容）。

### 插件做一切，内核只调度

OpenClaw 的内核非常克制。它自己不会调 OpenAI 的 API，不会连 Telegram，不会执行 bash 命令 -- 这些事全部交给插件做。内核只负责四件事：管理插件的生命周期、编排 Agent 的执行流程、路由消息到正确的目的地、管理配置和会话。

这种"精简内核 + 丰富插件"的设计，让系统可以不断增加能力而不用改核心代码。想支持一个新的聊天平台？写个 Channel 插件。想接入一个新的 AI 模型？写个 Provider 插件。想给 AI 加个新技能？写个 Tool 插件。

### 先看证件，再让进门

OpenClaw 对插件安全有一个独特的设计叫 Manifest-First。传统的插件系统是"先加载代码，跑起来之后才知道这个插件能干什么"。OpenClaw 反过来：每个插件必须带一个 JSON 声明文件，系统先读这个声明，校验格式、检查权限、确认不冲突，一切没问题之后才加载代码。

这意味着一个恶意插件如果声明文件不合格，它的代码根本不会被执行。就像进公司先刷工牌，不是先让你进来再查你是不是员工。

插件加载分四步：扫描发现所有声明文件、校验筛选出合格的、按依赖顺序加载代码、注册能力到全局注册表。之后系统各处（Agent 引擎、通道管理器、Hook 系统）从注册表里取自己需要的东西用。

### 通道是一等公民

大多数 AI 聊天产品把 Web UI 当主体，消息平台集成是附加功能。OpenClaw 反过来：消息通道是系统的核心概念，Web UI 只是其中一个通道。

每个通道插件是一组可组合的适配器：有的负责账号配置，有的负责消息发送，有的负责安全策略，有的负责群组管理，有的负责消息状态反馈。一个全功能的 Telegram 插件可能实现了 15 个适配器，一个简单的 Webhook 通道只需要实现 3 个。通道插件不需要实现所有能力，只实现自己支持的部分。

通道管理器在中间做翻译：入站时把各平台的消息格式统一成内部格式，出站时把内部格式转回各平台的格式。对 Agent 引擎来说，它完全不知道消息是从 Telegram 还是 Discord 来的，它只看到统一格式的消息。

### 安全是默认值，不是可选项

OpenClaw 跑在用户自己的机器上，能执行命令、访问文件，所以安全设计贯穿每一层：

Gateway 连接需要 Token 认证，没有 Token 连不上。新设备首次连接需要配对，配对需要管理员手动审批，审批用的是加密签名。DM 消息默认走配对模式，只有你主动放开才会接受陌生人消息。

AI 调用工具时有多层检查：先检查工具是否在白名单里，再检查输入是否合法，然后看这个工具是否需要放在沙箱里执行（比如执行用户写的脚本）。对于危险命令（比如 `rm -rf`），系统会拦截并要求管理员审批。

配置文件里的密钥不直接写明文，而是用引用的方式（比如 `${env.API_KEY}`），运行时才去环境变量或密钥库里取真实值。

### 用测试守住架构边界

百万行代码的项目，最怕的是架构腐化 -- 本来不该互相依赖的模块慢慢耦合在一起。OpenClaw 的做法是把架构规则写成自动化测试：如果有人在插件代码里 import 了内核的内部模块，CI 会直接报错。这些叫"架构气味测试"，不是测功能对不对，而是测结构有没有变形。

## Agent 引擎：AI 的大脑

Agent 引擎是系统里最复杂的部分，它负责让 AI "思考"和"行动"。

核心是一个循环：把用户消息和上下文发给 AI 模型，AI 要么直接回复文字，要么请求调用一个工具。如果调了工具，系统执行工具、拿到结果、再连同结果一起发回给 AI，让 AI 继续思考。这个循环直到 AI 决定不再调用工具、直接输出文字回复为止。

模型选择有一套优先级链：如果这个会话手动指定了模型就用指定的，否则看 Agent 配置，再看通道级别的覆盖，最后是全局默认。如果主模型调用失败（比如限流了），系统自动切到备用模型，备用也失败就切第二备用，全挂了才报错。

工具执行有前后 Hook 插入点。你可以写一个 Hook 在工具执行前拦截（比如记录日志或者弹审批），也可以在执行后修改结果。Hook 分同步和异步两种：同步 Hook 会阻塞流程（适合审批场景），异步 Hook 不影响主流程（适合日志记录）。

## 配置如何流动

OpenClaw 的配置来源有四个：YAML 配置文件、环境变量、SecretRef 引用（密钥不写明文）、CLI 参数。系统启动时把这四个来源合并成一份完整配置，中间会做格式校验和旧版迁移。

配置文件改了之后，系统不需要重启就能感知到。它用文件监听器检测变化，然后计算哪些字段改了。改了密钥或 Agent 列表这类安全的东西，直接热生效；改了插件配置或 Provider 认证这种涉及初始化流程的，才需要重启 Gateway。这是一种折中 -- 既不像"全热重载"那样有一致性风险，也不像"全重启"那样影响可用性。

会话数据存在磁盘上，就是 JSON 文件。每次对话结束后追加写入，用原子文件操作保证不会写坏。没有用数据库，因为自托管场景不想让用户额外装一个 PostgreSQL。

## 设备能力扩展：手机也是工具

OpenClaw 有一个独特的 Node 概念：你的 iPhone、Android 手机、Mac 电脑可以作为"设备节点"接入 Gateway。接入后，AI 就可以通过 `node_invoke` 工具遥控这些设备 -- 比如让手机截个屏、点击某个按钮、发条短信、拍张照。

设备通过 WebSocket 连到 Gateway，首次连接需要配对（显示配对码，管理员审批）。之后的每次通信都用加密签名验证身份。AI 发出设备命令后，Gateway 转发给对应设备，设备执行完把结果（包括截图）送回来，Gateway 再把结果交给 AI 继续推理。

## 值得借鉴的几个设计

**Manifest-First 插件发现。** 任何需要插件系统的项目都可以借鉴。核心思路：用声明式元数据替代命令式注册，在加载代码前完成安全校验。

**Per-Session 串行 + 跨 Session 并行。** 任何 AI Agent 系统都面临并发问题。OpenClaw 的做法很精确：同一对话严格排队保正确性，不同对话完全并行保吞吐量。

**架构气味测试。** 大型项目的架构规则不能只靠文档和 Code Review，要用自动化测试持续检测导入违规和层级穿透。

**混合热重载。** 长期运行的服务不能什么配置改了都重启，也不能什么都不重启。区分安全变更和结构性变更，前者热生效，后者才重启。

## 总结

OpenClaw 本质上在做一件事：**在你的各种聊天窗口和你的 AI 助手之间架一座桥**。

这座桥要解决三个问题：怎么把 20 多个聊天平台的消息统一起来（通道适配），怎么让 AI 不只是说话还能干活（Agent 引擎 + 工具循环），怎么保证跑在用户自己机器上的这套东西是安全的（全链路安全设计）。

它选择了一种"精简内核 + 丰富插件"的架构，内核只做调度，具体能力全交给插件。所有插件先过安全审查再加载代码。所有通信走一个 WebSocket 端口。所有状态存在本地文件里。没有数据库，没有微服务，没有云依赖 -- 就是一个进程，跑在你的机器上，听你的。
