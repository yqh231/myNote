## 1. 什么是上下文工程

### 1.1 定义

上下文工程 (Context Engineering) 是指**在每一次 LLM 推理调用前，以工程化、可复用、可度量的方式，拼装并优化输入上下文 (tokens)**，从而最大化获得期望输出的概率。

一句话总结：**Prompt Engineering 关注"怎么写指令"，Context Engineering 关注"怎么组装整个输入"**。

### 1.2 与 Prompt Engineering 的关系

| 维度 | Prompt Engineering | Context Engineering |
|------|-------------------|-------------------|
| 关注点 | 系统提示的措辞与结构 | 整个上下文窗口的 tokens 配置 |
| 覆盖范围 | 单轮/少量轮次的指令优化 | 跨多轮推理的状态管理 |
| 管理对象 | 系统提示、用户提示 | 系统指令 + 工具定义 + 外部数据 + 消息历史 + 记忆 |
| 适用阶段 | 早期原型与简单用例 | 生产级 Agent 与长时程任务 |

上下文工程是 Prompt Engineering 的自然演进。当我们从"单轮问答"走向"循环运行的 Agent"时，仅靠写好提示词远远不够，需要管理**整个上下文状态**。

### 1.3 为什么重要

**核心矛盾：上下文是有限资源，且存在边际收益递减。**

- **上下文腐蚀 (Context Rot)**：随着上下文窗口中 tokens 增加，模型从中准确回忆信息的能力反而下降。Needle-in-a-Haystack 测试反复验证了这一点。
- **注意力预算**：Transformer 的 n^2 注意力机制意味着，上下文越长，注意力被"拉薄"得越多。
- **训练分布偏差**：短序列在训练数据中更常见，模型对超长上下文的处理能力天然弱于短上下文。

因此，**每多放一个 token 进上下文，都必须确保它的价值大于其"注意力成本"**。

## 2. 上下文工程的操作框架

### 2.1 GSSC 流水线

业界主流的上下文构建方法论可归纳为 **GSSC 四阶段流水线**，这也是 HelloAgents 框架 ContextBuilder 的核心设计：

```
Gather (收集) --> Select (选择) --> Structure (结构化) --> Compress (压缩)
```

#### 阶段一：Gather - 多源信息收集

从所有可用来源收集候选信息，将其统一封装为标准化的信息包 (ContextPacket)。

**典型信息来源及优先级**：

| 优先级 | 来源 | 说明 |
|--------|------|------|
| P0 | 系统指令 (System Prompt) | 角色定义、行为约束、输出格式要求 |
| P1 | 任务状态与记忆 (Memory) | 当前子目标、关键结论、阻塞项 |
| P2 | 事实证据 (RAG/知识库) | 与查询相关的外部知识检索结果 |
| P3 | 对话历史 (History) | 最近 N 轮对话，提供交互上下文 |
| P4 | 工具输出 (Tool Results) | 最近一次工具调用的结果 |

**关键设计原则**：

- **容错性**：每个数据源的调用都应独立 try-except，单个源失败不影响整体。HelloAgents 的 `_gather()` 方法正是这样做的：

```python
# 每个来源独立处理，互不影响
if self.memory_tool:
    try:
        state_results = self.memory_tool.execute("search", ...)
    except Exception as e:
        print(f"记忆检索失败: {e}")  # 降级而非崩溃

if self.rag_tool:
    try:
        rag_results = self.rag_tool.run({"action": "search", ...})
    except Exception as e:
        print(f"RAG检索失败: {e}")
```

- **统一封装**：不同来源的信息统一为 `ContextPacket`，携带 content、timestamp、token_count、relevance_score 和 metadata，便于后续统一评分和选择。

#### 阶段二：Select - 智能信息筛选

从候选池中选出最有价值的信息子集，核心是**在有限 token 预算内最大化信息价值**。

**评分公式**：

```
综合分 = w_relevance * 相关性分数 + w_recency * 新近性分数
```

默认权重：相关性 0.7，新近性 0.3。

**相关性计算**：

- 基础方案：关键词重叠 / Jaccard 相似度（HelloAgents 默认实现）
- 生产方案：向量 Embedding 余弦相似度、BM25 + 向量混合检索

**新近性计算 - 指数衰减模型**：

```python
recency_score = exp(-delta / tau)
# delta: 距今时间(秒)
# tau: 衰减时间尺度(如 3600 秒 = 1小时)
```

**选择策略**：

1. 系统指令**无条件保留**（不参与评分竞争）
2. 过滤掉相关性低于 min_relevance 阈值的信息
3. 按综合分降序，**贪心填充**直至 token 预算耗尽

**业界进阶做法 - MMR (最大边际相关性)**：

HelloAgents 的 ContextConfig 中预留了 `enable_mmr` 和 `mmr_lambda` 参数。MMR 在选择时同时考虑**相关性**和**多样性**，避免选入大量高度重复的信息：

```
MMR_score = lambda * sim(doc, query) - (1-lambda) * max(sim(doc, selected_docs))
```

#### 阶段三：Structure - 结构化组织

将选中的信息按固定骨架组织，而非简单拼接。HelloAgents 采用的分区模板：

```
[Role & Policies]    -- 系统指令：角色定位与行为约束
[Task]               -- 当前任务描述
[State]              -- 任务状态：进展、未决问题
[Evidence]           -- 事实证据：RAG检索结果、知识库引用
[Context]            -- 辅助材料：对话历史、相关记忆
[Output]             -- 输出约束：格式要求、回答结构
```

**为什么要结构化？**

- 模型更容易定位和利用不同类型的信息
- 便于调试：可以快速定位哪个分区的信息有问题
- 便于 A/B 测试：可以单独替换某个分区的内容

**业界通用做法**：使用 XML 标签或 Markdown 标题作为分隔符，例如 Anthropic 推荐使用 `<instructions>`、`<context>`、`<output_format>` 等 XML 标签。

#### 阶段四：Compress - 预算守护与压缩

当结构化后的上下文仍超预算时，执行兜底压缩：

```python
# HelloAgents 的压缩策略
available_tokens = max_tokens * (1 - reserve_ratio)  # 预留 15-20% 生成余量

if current_tokens > available_tokens:
    # 按段落逐行截断，保留结构完整性
    for line in lines:
        if used_tokens + line_tokens > available_tokens:
            break
        compressed_lines.append(line)
```

**业界压缩方法梯度**：

| 方法 | 保真度 | 成本 | 适用场景 |
|------|--------|------|----------|
| 简单截断 | 低 | 零 | 兜底方案 |
| 按段落截断（保留结构） | 中 | 零 | HelloAgents 默认方案 |
| LLM 摘要压缩 | 高 | 额外 API 调用 | 生产环境 |
| LLMLingua 语义压缩 | 高 | 推理开销 | 高精度场景 |
| 清理深层历史中的工具调用 | 中 | 零 | 长对话场景 |

### 2.2 Token 预算管理

一个常被忽视但至关重要的实践：**为输出预留 token 空间**。

```python
# HelloAgents 的做法
reserve_ratio = 0.15  # 预留 15% 给模型生成
available = max_tokens * (1 - reserve_ratio)
```

**经验数据**：

- 简单问答：预留 10%
- 代码生成：预留 20-30%
- 长文档生成：预留 40% 以上

### 2.3 KV-Cache 优化 (业界前沿)

2025 年 Manus 团队分享的重要实践 -- **通过稳定上下文前缀最大化 KV-Cache 命中率**：

- **保持前缀稳定**：系统提示 + 工具定义 放在最前面且不变，避免动态元素（如时间戳）破坏缓存
- **追加式更新**：新信息追加到上下文末尾，而非插入中间
- **确定性序列化**：工具定义使用固定 JSON key 顺序
- **逻辑屏蔽而非物理移除**：临时不用的工具用 logit masking 禁用，而非从上下文中删除

这种优化在高并发场景下可以显著降低推理成本和延迟。

## 3. 长时程任务的三大策略

当任务跨度超过单次上下文窗口时，需要以下工程手段：

### 3.1 压缩整合 (Compaction)

当对话接近上下文上限时，对历史内容进行**高保真摘要**，携带摘要重启新的上下文窗口。

**操作要点**：

- 保留：架构决策、未解决缺陷、实现细节、关键结论
- 丢弃：重复的工具输出、冗余的中间推理步骤
- 调优：先保召回（不遗漏），再提精度（去冗余）

**适用场景**：需要长对话连续性的任务，强调上下文的"接力"。

### 3.2 结构化笔记 (Structured Note-taking)

Agent 定期将关键信息写入**上下文外的持久化存储**，后续按需拉回。

**典型实现**：

- 维护 `TODO.md`、`NOTES.md`、`PROGRESS.md` 等文件
- 记录子目标完成状态、关键结论、依赖关系、阻塞项
- 跨多轮上下文重置仍能保持进度一致性

HelloAgents 提供了配套的 `NoteTool`，以 Markdown + YAML 前置元数据的格式管理笔记，兼顾人类可读性和版本控制友好性。

**适用场景**：有里程碑的迭代式开发与研究任务。

### 3.3 子代理架构 (Sub-agent Architectures)

主代理负责高层规划与综合，子代理在**独立的干净上下文窗口**中各自深挖，最后只回传**凝练摘要**（通常 1,000-2,000 tokens）。

```
主代理 (规划与综合)
  |-- 子代理A (搜索文档) --> 返回摘要
  |-- 子代理B (执行代码) --> 返回结果
  |-- 子代理C (分析数据) --> 返回结论
```

**核心价值**：关注点分离。庞杂的搜索/执行上下文留在子代理内部，主代理专注整合与推理。

**适用场景**：复杂研究与分析，能从并行探索中获益。

## 4. 实践案例

### 4.1 案例一：ContextBuilder 构建客服 Agent 上下文

以下展示 HelloAgents 的 ContextBuilder 如何为一个数据分析顾问 Agent 构建上下文：

```python
from hello_agents.context import ContextBuilder, ContextConfig

# 配置：8000 token 预算，15% 预留给生成
config = ContextConfig(max_tokens=8000, reserve_ratio=0.15)
builder = ContextBuilder(
    memory_tool=memory_tool,
    rag_tool=rag_tool,
    config=config
)

# 一次调用完成 GSSC 四阶段
context = builder.build(
    user_query="如何优化Pandas的内存占用?",
    conversation_history=history,
    system_instructions="你是一位资深的Python数据工程顾问..."
)
```

**产出的结构化上下文**：

```
[Role & Policies]
你是一位资深的Python数据工程顾问。你的回答需要:
1) 提供具体可行的建议 2) 解释技术原理 3) 给出代码示例

[Task]
如何优化Pandas的内存占用?

[Evidence]
Pandas内存优化的核心策略包括:
1. 使用合适的数据类型(如category代替object)
2. 分块读取大文件 ...

[Context]
user: 我正在开发一个数据分析工具
assistant: 很好!您计划使用什么技术栈?
记忆: 用户正在开发数据分析工具,使用Python和Pandas
记忆: 已完成CSV读取模块的开发

[Output]
请基于以上信息,提供准确、有据的回答。
```

**分析**：ContextBuilder 自动完成了多源信息收集（记忆 + RAG + 历史）、相关性评分、结构化组织和预算守护，开发者无需在每个 Agent 中重复编写这些逻辑。

### 4.2 案例二：JIT 上下文 vs 预加载检索

**传统 RAG 方式（预加载）**：

```
用户提问 --> Embedding 检索 --> 一次性加载所有相关文档 --> 塞入上下文 --> LLM 推理
```

问题：可能加载了大量"看起来相关但实际没用"的文档，浪费 token 预算。

**JIT (Just-in-Time) 上下文方式**：

```
用户提问 --> Agent 分析需求
         --> 决定需要哪些信息
         --> 按需调用工具获取（grep、文件读取、API 查询）
         --> 只将必要结果纳入上下文
         --> LLM 推理
```

**典型实现**：Claude Code 的做法是预加载项目约定文件（如 CLAUDE.md），然后提供 glob、grep、read 等工具让 Agent 按需检索具体文件，而非预先索引整个代码库。

**混合策略（推荐）**：

```python
# 预加载高价值上下文（快速启动）
context = load_project_conventions()  # README、规范文档等

# Agent 运行时按需检索（精确补充）
if agent.needs_more_context():
    result = agent.use_tool("grep", pattern="def calculate_tax")
    context.append(result)  # 只追加必要信息
```

### 4.3 案例三：长时程代码迁移任务

假设需要将一个大型项目从 Python 2 迁移到 Python 3，任务跨度数小时：

**阶段一 - 规划（子代理扫描）**：

```
主代理: "分析项目结构，识别所有需要迁移的文件"
  子代理A: 扫描 print 语句 --> "发现 127 个文件使用 print 语句"
  子代理B: 扫描 unicode 处理 --> "发现 43 个文件有 unicode 兼容问题"
  子代理C: 扫描依赖 --> "3 个依赖库不支持 Python 3"
```

**阶段二 - 执行（结构化笔记 + 压缩整合）**：

```markdown
<!-- PROGRESS.md - Agent 维护的笔记 -->
## 迁移进度
- [x] print 语句迁移 (127/127)
- [ ] unicode 处理 (18/43)  <-- 当前阶段
- [ ] 依赖升级 (0/3)

## 阻塞项
- mysql-python 库无 Python 3 版本，需替换为 mysqlclient

## 关键决策
- 统一使用 f-string 替代 % 格式化和 .format()
```

每当对话接近上下文上限，执行 Compaction：将已完成的迁移细节压缩为"print 语句迁移已全部完成"，只保留未完成项和阻塞项的详细信息。

## 5. 最佳实践清单

### 系统提示设计

- 分区组织（用 XML 标签或 Markdown 标题），追求"最小必要信息集"
- 先用最好的模型在最小提示上试跑，根据失败模式逐步增补
- 避免两个极端：过度硬编码 if-else 逻辑 / 过于空泛只给宏观目标

### 工具设计

- 职责单一、接口语义清晰、相互低重叠
- 返回 token 友好的精简结果，而非原始大块数据
- **如果人类工程师都说不清用哪个工具，别指望 Agent 做得更好**

### 信息选择

- 将上下文视为"注意力预算"，每个 token 都有成本
- 相关性 + 新近性加权评分是简单有效的基线方案
- 生产环境用向量检索替代关键词匹配提升检索质量
- 考虑 MMR 兼顾多样性，避免信息冗余

### 监控与迭代

- 记录每次上下文构建的统计信息（token 使用率、信息源命中率）
- 对关键参数（权重、阈值）做 A/B 测试
- 保留错误日志在上下文中，帮助 Agent 自适应学习

---

## 6. 总结

上下文工程的核心思想可以浓缩为一句话：

> **在有限的注意力预算内，用尽可能少但信号密度高的 tokens，最大化模型产出期望结果的概率。**

操作上遵循 GSSC 流水线：**收集**多源候选信息 --> **选择**高价值子集 --> **结构化**组织为模板 --> **压缩**守护 token 预算。对于长时程任务，配合压缩整合、结构化笔记和子代理架构三大策略。

即便模型能力持续提升、上下文窗口不断扩大，"在长交互中维持连贯性与聚焦"仍是构建强健 Agent 的核心挑战。谨慎而系统的上下文工程将长期保持其关键价值。
