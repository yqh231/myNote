# Claude Code Agent 设计精要（下）: 工程细节与代码检索

基于 Claude Code 泄露源码的深度阅读，提炼 Agent 系统设计中值得沉淀的技术点。聚焦"为什么这样设计"和"解决了什么真实工程问题"。

下篇聚焦工程实现细节（编译安全、流式撤回、幂等写入）和代码检索全链路（三层检索协作、结果裁剪预算、模型与工程的分工边界）。

---

## 7. 编译期特性开关: 真正的零成本

Claude Code 同时服务内部版和外部版。内部有实验功能、调试工具、内部 API 集成。这些代码不能出现在外部产物里。

传统做法是 `if (process.env.INTERNAL) { ... }`，但这有运行时开销（分支预测），且代码仍然在产物中（可以被反编译看到）。

Claude Code 用 Bun 打包器的 `feature()` 宏实现**编译期消除**:

```
feature('COORDINATOR_MODE') 为 false 时:
  -> require('./coordinator/...') 整个 ternary 在编译时消失
  -> coordinator 目录下的所有代码都不会被打包
  -> 产物中连字符串 'COORDINATOR_MODE' 都不存在
```

但这有个严格的约束: `feature()` 只能直接出现在 `if` 或三元表达式中。如果先赋值给变量再判断，Bun 就无法在编译期分析了，代码不会被消除。更微妙的是，feature gate 保护的模块里的字符串也会被消除 -- 在边界处多加一行代码都可能破坏整个消除链。

## 8. Tombstone: 流式系统中的"撤回"

模型 streaming 过程中发生了降级（切到备用模型），已经发出去的 assistant 消息变成了"孤儿" -- 签名对新模型无效，不能留。

但消息已经被 UI 渲染了，已经被 SDK 消费者接收了。不能假装没发过。

方案: 发一个 Tombstone 消息，语义是"之前那条消息作废了"。消费者收到后从渲染中移除、在历史中标记为无效。这比"不发"更安全 -- 消费者手里已经有了原始消息，必须显式告知其失效。

## 9. 幂等写入: 一个 flag 解决竞态

同一个 tool_result 可能在多次 API 调用中被处理（压缩回放、会话恢复），但只需要写入磁盘一次。

朴素做法: 先检查文件是否存在（`stat`），不存在才写入（`writeFile`）。问题: stat 和 writeFile 之间有个时间窗口，两个并发请求可能都 stat 了"不存在"然后都写入。

Claude Code 用了一个 `wx` flag（write + exclusive）: 写入时如果文件已存在就报错 `EEXIST`。一步原子操作，没有竞态窗口。收到 EEXIST 不算错，只是说明之前已经写过了，直接跳过。

## 10. 代码检索: 不完全是模型在搜，工程侧做了三层铺垫

检索不是单纯"模型调搜索工具"，而是工程代码和模型的三层协作。

### 10.1 第一层: 工程侧自动注入（模型被动接收，不用搜就能看到）

模型发第一个请求之前，工程代码已经准备好了一批上下文:

**CLAUDE.md 层级加载**（`src/utils/claudemd.ts`）: 从 CWD 往上遍历每一层目录，自动发现并拼接所有 CLAUDE.md、`.claude/rules/*.md`、CLAUDE.local.md。支持 `@path` 指令引用其他文件。优先级从低到高: 全局管理员指令 -> 用户个人指令 -> 项目指令 -> 本地私有指令。

**Git 状态快照**（`src/context.ts`）: 会话开始时并行获取当前分支、主分支名、`git status --short`（截断到 2000 字符）、最近 5 条 commit、git 用户名。注入后不再更新 -- 是快照不是实时状态。

**IDE 上下文注入**（`src/utils/attachments.ts`）: 在 VS Code 中使用时，每次用户发消息自动附加当前打开的文件名、用户选中的代码、编译错误/警告、LSP 诊断信息。这就是为什么在 IDE 里打开一个文件然后问 Claude，它"好像知道"你在看什么 -- 不是模型搜到的，是工程代码自动注入的。

### 10.2 第二层: 工程侧增强的搜索工具（模型主动调用，工程侧深度优化）

模型确实需要主动调用 Grep/Glob/LSP，但这些工具不是简单的 wrapper:

**Grep（内容搜索）**: 底层调 ripgrep，自动排除 `.git` 等目录、限制单行 500 字符（防 base64 刷屏）、默认只返回前 250 条、路径自动转相对路径（省 token）、按修改时间排序、支持分页。

**Glob（文件名搜索）**: 模式匹配找文件，硬上限 100 个，按修改时间排序。

**LSP（代码智能）**: 跳转定义、查找引用、查看类型。知道函数名时比 Grep 精准得多。

### 10.3 第三层: 纯模型驱动的检索策略

System Prompt 不只给工具，还教了**使用方法论**:

- **投机式并行搜索**: 一口气发多个搜索，由于 Grep/Glob 都是 `isConcurrencySafe: true`，它们会真正并行执行
- **先搜索再动手**: 被明确要求先调研再编码
- **知道什么时候升级**: Grep 搜不到就启动 Explore 子 Agent 做深度探索，搜索结果不塞进主 Agent 上下文

但**搜什么关键词、用哪个工具、读哪个文件、什么时候搜索够了** -- 这些决策完全由模型自主完成。

### 10.4 没有 RAG，没有索引

从源码确认: **没有嵌入向量、没有预建索引、没有 RAG 管线、没有语法树分析**。搜索就是暴力的 ripgrep + 模型的智能判断。

这是一个重要的设计决策 -- 零启动成本、零维护成本、对任何语言都有效。结果限流重于搜索精度: 宁可只给 250 条让模型自己判断，也不做复杂的相关性排序，因为模型本身就是最好的相关性判断器。

### 10.5 超大代码库的效率处理

ripgrep 天生快（Rust 实现、并行扫描、自动跳 `.gitignore`）。结果限流防上下文爆炸。路径压缩省 token。分页而非全量 -- 模型自己决定要不要看更多。

最关键的是**子 Agent 隔离上下文**: 大量探索 spawn 到 Explore 子 Agent，搜索中间过程不进入主 Agent 上下文，只返回精炼结论。

### 10.6 Tool 延迟加载: 60+ 个 Tool 不全塞进 Prompt

这是一个很巧的间接优化。Claude Code 有 60+ 个 Tool，如果把所有 Tool 的 JSON Schema 都放进 System Prompt，光这些 Schema 就要吃掉大量 token。

解决方案: **ToolSearch 机制**。只有核心 Tool（Grep、Glob、Read、Edit、Bash、Agent 等）的 Schema 放在初始 Prompt 里。MCP 扩展工具和不常用 Tool 标记为 `shouldDefer: true`，初始 Prompt 里只出现名字（不带 Schema）。

当模型需要用某个延迟工具时，先调 `ToolSearch` 搜索，ToolSearch 返回该工具的完整 Schema，模型才能调用它。这个机制:

- 初始 Prompt 省了大量 token（据统计约 10% 的全局 token 消耗）
- 不常用的工具不浪费注意力
- MCP 扩展工具自动延迟加载，不管接了多少 MCP 服务器都不会撑爆 Prompt

ToolSearch 本身支持两种查询方式:
- `select:Read,Edit,Grep` -- 直接按名字选，精确匹配
- `notebook jupyter` -- 关键词搜索，按工具名称、searchHint、描述的匹配程度打分排序

### 10.7 实际检索流程示例

用户说: "给登录页面加一个记住密码的功能"

模型的典型检索过程:

```
第1步: 并行搜索（多个 Tool 同时跑）
  Glob("**/login*")       -> 找到 login 相关的文件
  Grep("login|sign.?in")  -> 找到包含 login 的代码
  Grep("remember|password")-> 看看有没有已有的记住密码逻辑

第2步: 聚焦阅读
  Read("src/pages/Login.tsx")  -> 读登录页面组件
  Read("src/api/auth.ts")     -> 读认证 API

第3步: 如果结构不清楚，用 LSP
  goToDefinition("handleLogin") -> 找到 handleLogin 的定义
  findReferences("AuthContext")  -> 看认证上下文在哪被用到

第4步: 如果代码库很大、关系复杂
  Agent(Explore, "分析这个项目的认证架构") -> 子 Agent 做深度调研
  子 Agent 内部可能做 20+ 次搜索和读取
  最终返回一段结论性的架构描述

第5步: 基于理解开始改代码
```

整个过程的关键原则: **先粗后细，先面后点，搜索和阅读并行，大量探索隔离到子 Agent**。

## 11. Tool 结果裁剪: 搜到之后喂给模型多少

检索到文件后，"哪些内容进入模型上下文"由工程代码三层预算自动控制，模型只能在预算框架内做有限的主动决策。

### 11.1 工具级限流: 每个工具管自己的输出大小

- **Read**: 默认最多读 2000 行，文件超过 256KB 或输出超过 25000 token 直接报错（不截断）。模型必须用 offset/limit 分段重读
- **Grep**: 默认只返回前 250 条匹配，单行最长 500 字符，支持 offset 翻页
- **Glob**: 硬限 100 个文件

Read 超限时选择报错而非截断。源码注释: 曾经试过截断，error rate 下降但 mean tokens 反而上升 -- 报错只消耗 ~100 byte，截断则产生 ~25K token。报错迫使模型用更精确的范围重读，反而更省上下文。

### 11.2 单结果落盘: 超大结果存磁盘，只给模型看预览

单个 tool_result 超过阈值（默认 50000 字符）时，整个结果写入磁盘，模型只看到前 2KB 预览 + 文件路径。需要完整内容时可用 Read 读回。每个工具可声明自己的阈值（Read 设为 Infinity -- 因为 Read 输出再存文件让模型用 Read 读回来是循环的）。

### 11.3 消息级预算: 同一轮多个工具结果的总和限流

模型可能一次调 10 个并行工具，每个返回 40K，合计 400K。消息级预算: 同一条 user 消息内所有 tool_result 合计超过 200K 字符时，从最大的结果开始逐个落盘替换为预览。

这个预算与决策冻结联动（见上篇第 3 节）-- 一个 tool_result 第一次发送给 API 时是"原文"还是"预览"就冻结了，以后每次都用完全一样的内容，保证 Prompt Cache 一致。

### 11.4 模型怎么决定 Read 的 offset 和 limit

模型不是"智能"地决定读哪段，而是靠工具返回的元数据和错误信息做简单推理:

**信息来源一: 返回结果的元数据**。Read 每次返回时带 `totalLines`（文件总行数）、`startLine`（起始行）、`numLines`（本次返回行数）。模型看到"文件 8500 行但只读了 2000 行"，就知道后面还有内容，自己算出下一段的 offset。

**信息来源二: 报错信息直接教模型怎么做**。超 token 限制时返回: "File content (35000 tokens) exceeds maximum allowed tokens (25000). Use offset and limit parameters to read specific portions of the file." 错误文本里明确写了该用什么参数。

**信息来源三: System Prompt 的工具描述**。模型一开始就知道 "By default, it reads up to 2000 lines" 以及 "You can optionally specify a line offset and limit"。

**最聪明的用法: 先搜后跳**。模型通常不会盲目翻页，而是先用 Grep 定位到目标行号附近，再用 offset 精确跳转。比如 Grep 发现目标在第 5000 行，就直接 `offset: 4990, limit: 50`。这不是工程代码教的，是模型自己从上下文推理出来的策略。

## 下篇总结

| 挑战 | Claude Code 的应对 | 本质 |
|------|-------------------|------|
| 编译安全 | 编译期 feature gate + DCE | 内部代码在外部产物中物理不存在 |
| 流式撤回 | Tombstone 消息 | 已发出的消息不能假装没发，要显式作废 |
| 幂等写入 | `wx` flag 原子操作 | 一步解决竞态，EEXIST 不是错误是正常 |
| 代码检索 | 三层协作: 自动注入 + 增强工具 + 模型策略 | 没有 RAG/索引，暴力搜索 + 模型智能判断 |
| 结果裁剪 | 工具级/单结果/消息级三层预算 | 报错优于截断，落盘优于丢弃 |
| 模型读文件 | 元数据 + 错误信息 + 先搜后跳 | 模型靠信息推理，不靠"智能" |

上下两篇合在一起，覆盖了 Claude Code 作为生产级 Agent 系统的核心设计。这些不是教科书上的设计模式，而是在大规模生产环境中踩过坑之后沉淀出来的。每一个都对应着一个真实的生产事故或性能问题。
