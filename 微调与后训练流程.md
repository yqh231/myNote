# LLM 微调与后训练: 从原理到前沿

> 来源: *Build a Large Language Model From Scratch* (Sebastian Raschka) 分类微调与指令微调章节, 结合业界实践与前沿进展。

---

## 一、后训练的本质: 预训练到底缺了什么

### 1.1 预训练模型的"能力"与"行为"分离

预训练完成后, 模型已经**掌握了几乎所有能力**: 语言理解、世界知识、逻辑推理的雏形。但它有一个根本性的问题 -- **它不知道自己该做什么**。

给一个预训练 GPT 输入 "What is the capital of France?", 它不会回答 "Paris", 而是续写出 "What is the capital of Germany? What is the capital of Italy?..." -- 因为在预训练语料中, 这种问句后面最可能跟的就是更多问句。

这揭示了一个深刻的认知: **预训练建立的是概率分布上的"能力空间", 但没有在这个空间中选定一个"行为策略"**。后训练的全部意义, 就是在这个能力空间中**挑选并强化一种特定的行为模式**。

用一个类比: 预训练像是一个人读了全世界的书, 但从来没有和人对话过。他什么都"知道", 但不知道别人问他问题时应该回答, 而不是背诵更多书。

### 1.2 后训练的阶段划分

```
预训练 (Pretraining)  -- 建立能力空间
  |
  v
监督微调 SFT           -- 选定行为模式: "被问就答"
  |
  v
偏好对齐 RLHF/DPO      -- 优化行为质量: "答得好"
  |
  v
可选: RL 强化 (GRPO等)  -- 涌现高阶能力: "会推理"
  |
  v
可选: 领域/任务特化      -- 收窄能力: "专精某领域"
```

书中涵盖了两种微调: **分类微调**(任务特化的极端形式)和**指令微调**(SFT)。它们虽然都叫"微调", 但在后训练 pipeline 中的位置和本质完全不同。

## 二、分类微调: 不是"微调", 而是"改造"

### 2.1 本质: 从生成模型到判别模型的范式转换

分类微调做的事情远比"调整参数"激进 -- 它**改变了模型的计算范式**:

```
预训练 GPT: 输入序列 -> 预测下一个词 (生成式, 输出空间 = 词表大小 50257)
分类微调后: 输入序列 -> 预测类别标签 (判别式, 输出空间 = 类别数 2)
```

具体做法是砍掉原始的 `Linear(768, 50257)` 输出头, 换上 `Linear(768, 2)` 分类头。这不是"微调", 这是**手术** -- 模型的"嘴"被换掉了。

**为什么这样做有效?** 关键洞察是: 预训练过程中, 为了准确预测下一个词, transformer 的中间层必须学会**理解文本的语义、情感、意图**。这些理解能力编码在 hidden state 中, 而 hidden state 对分类和对生成同样有用。分类微调本质上是**复用预训练建立的表示空间, 但改变了表示的"读出方式"**。

### 2.2 GPT 做分类的独特挑战: 信息流方向问题

BERT 用 `[CLS]` token 配合双向 attention, 天然就有一个"全局摘要"位置。GPT 的 causal attention mask 使得信息只能从左往右流:

```
token_1: 只能看到 [token_1]
token_2: 能看到   [token_1, token_2]
token_3: 能看到   [token_1, token_2, token_3]
...
token_n: 能看到   [token_1, token_2, ..., token_n]  <- 唯一的"全局视野"位置
```

所以必须用最后一个 token 的 hidden state 做分类。这不是一个工程 trick, 而是 **causal attention 的数学必然** -- 只有序列末尾是信息的汇聚点。

这个设计有一个有趣的推论: **padding token 不能放在序列末尾之后用于分类**。因为 padding 是无意义的, 它的 hidden state 不携带有用信息。书中的做法是 pad 在末尾但取最后一个"真实 token"的位置。实际上更常见的做法是直接 pad, 因为 attention 会学会忽略 padding。但这里揭示了一个更深的问题: **在 decoder-only 架构上做判别任务, 天然比 encoder 架构 (BERT) 别扭**, 这也是为什么业界分类任务至今仍然大量使用 BERT 系列而非 GPT 系列。

### 2.3 冻结策略背后的表示空间理论

书中冻结大部分参数, 只解冻最后一个 transformer block + 分类头。为什么?

可以从**表示空间的几何**角度理解:

- **底层** (embedding + 前几层): 学习通用的语言特征 -- 词法、句法、基础语义。这些特征对几乎所有 NLP 任务通用, 不需要改。
- **顶层** (最后几层): 学习任务相关的高阶特征。预训练时它们被优化为"预测下一个词", 现在需要"旋转"到"判断类别"的方向。
- **分类头**: 全新的线性层, 从零学习从 hidden state 到类别的映射。

**为什么不只训练分类头 (Linear Probing)?** 因为预训练最后一层的 hidden state 是为 next-token prediction 优化的几何空间, 在这个空间中 "spam" 和 "ham" 的表示可能并没有被线性分开。解冻最后一层让模型有机会重组这个空间, 使两个类别变得线性可分。

这其实回答了一个更根本的问题: **迁移学习为什么有效?** 因为深度网络的底层特征具有跨任务的通用性 (universality), 而顶层特征具有任务特异性 (task-specificity)。微调本质上是在保留通用基础的同时, 调整任务特异的部分。

### 2.4 分类微调在今天还有意义吗?

**书中的方式 (替换输出头 + 微调) 在工业界已经不是主流做法了。** 当前的替代方案:

| 方法 | 思路 | 优劣 |
|---|---|---|
| **Prompt-based 分类** | 直接给 LLM 一个 prompt: "这条短信是 spam 吗? 回答 yes/no" | 零成本, 但准确率依赖 prompt 质量和模型能力, 对小模型效果差 |
| **Embedding + 轻量分类器** | 用 LLM 提取 embedding, 接 SVM / LightGBM / 小型 MLP | 推理快, 可解释性好, 适合大规模部署 |
| **LoRA 微调** | 不换输出头, 在 transformer 层注入低秩适配矩阵 | 比全量微调节省 10x+ 显存, 效果接近, 当前主流 |
| **BERT 系列** | 用 BERT/RoBERTa/DeBERTa 做分类 | 模型小, 推理快, 分类场景下性能不逊于 GPT, **依然是生产环境首选** |

**一个反直觉的事实**: 对于纯分类任务, 2024-2025 年的工业界依然大量使用 BERT 系列, 而不是 GPT 系列。原因: BERT 的双向 attention 天然更适合理解任务, 模型小推理快, 且在分类 benchmark 上并不弱于 decoder-only 模型。GPT 做分类的主要场景是: 已经有一个部署好的 LLM, 顺便让它做分类, 避免多模型维护成本。

## 三、指令微调 (SFT): 后训练的核心一步

### 3.1 SFT 的本质: 条件概率分布的重塑

从数学角度看, 预训练和 SFT 优化的都是 next-token prediction 的交叉熵:

```
L = -sum_t log P(x_{t+1} | x_1, ..., x_t; theta)
```

**损失函数完全相同**, 那为什么效果截然不同?

差异在于**训练数据定义了不同的条件概率分布**:

- 预训练: 数据是自然文本, 模型学到的是 P(next_word | context), 即"自然语言的续写模式"
- SFT: 数据是 instruction-response 对, 模型学到的是 P(response_word | instruction + response_prefix), 即"给定指令后的回答模式"

**SFT 没有改变模型的优化目标, 而是改变了模型的"经验分布"。** 模型依然在做 next-token prediction, 但它看到的"世界"变了 -- 从"互联网上的所有文本"变成了"人类提问 + 高质量回答"。在这个新的经验分布上做梯度下降, 模型的条件概率分布自然向"收到指令就给出回答"的方向偏移。

这就是为什么 SFT 数据量远小于预训练数据 (935 条 vs 数万亿 token), 却能产生质的行为变化: **它不需要让模型"学会"新东西, 只需要在已有的能力空间中"选择"一个行为子空间。** 就像一个会 100 种语言的翻译, 你只需要告诉他"用中文回答"就够了, 不需要重新教他中文。

### 3.2 "SFT 只是行为校准" 的边界在哪里?

"SFT 不教知识, 只教行为"这个说法被广泛引用, 但需要审慎看待:

**支持这个观点的证据:**
- LIMA 论文 (2023) 用仅 1000 条高质量数据微调 LLaMA-65B, 效果接近 GPT-4 (当时), 说明少量数据就能完成行为校准
- Alpaca (2023) 用 52K 条 GPT-4 生成的数据微调 LLaMA-7B, 效果不错, 说明 SFT 主要依赖预训练的知识基础

**挑战这个观点的证据:**
- 在领域特化 SFT 中 (如医学、法律), 模型确实能通过 SFT 数据学到预训练中没有的领域知识
- 代码 SFT 数据中包含的编程模式和 API 用法, 确实能被模型"学到"
- 多轮对话 SFT 教会了模型"记住上文"的能力, 这不仅是行为模式, 也是一种新能力

**更准确的说法是: SFT 主要是行为校准, 但在数据充足且预训练覆盖不到的领域, 它也能注入有限的新知识。** 行为校准是"快变量"(几百条数据就见效), 知识注入是"慢变量"(需要更多数据和更多 epoch)。

### 3.3 数据工程: 决定 SFT 成败的关键

#### Prompt Template: 不仅是格式, 更是"行为锚点"

书中使用 Alpaca 模板:

```
Below is an instruction that describes a task. Write a response that
appropriately completes the request.

### Instruction:
{instruction}

### Input:
{input}

### Response:
{output}
```

模板的作用远比"格式化数据"重要。从模型的角度看, `### Response:\n` 这个固定 token 序列是一个**条件触发器**: 模型学到的不是"一般性地生成好文本", 而是"看到这个特定 token 序列后, 切换到回答模式"。这就是为什么**推理时必须用和训练时完全一致的模板**, 哪怕少一个换行符都可能导致行为退化。

**业界模板演进及其背后的设计思考:**

| 代际 | 格式 | 解决什么问题 |
|---|---|---|
| Alpaca (2023) | `### Instruction / ### Response` | 最基本的指令遵循 |
| ChatML (OpenAI) | `<\|im_start\|>system/user/assistant<\|im_end\|>` | 多角色 + 多轮对话 + system prompt |
| Llama 3 (2024) | `<\|begin_of_text\|><\|start_header_id\|>...<\|end_header_id\|>` | 结构化 header, 支持工具调用 |
| 通用趋势 | 使用**特殊 token** 而非自然语言标记 | 避免和正文混淆, 更鲁棒 |

一个重要的工程启示: **选用特殊 token (如 `<|im_start|>`) 而非自然语言标记 (如 `### Response:`) 作为行为锚点, 是更好的设计。** 因为自然语言标记可能在正文中出现, 造成意外的模式切换; 而特殊 token 在预训练语料中从未出现过, 是一张"白纸", 可以被赋予精确的语义。

#### Loss Masking: 被低估的关键技术

书中的 `custom_collate_fn` 做了 padding mask (用 `ignore_index=-100`), 但提到了一个没有实现的更重要的优化: **instruction mask**。

```
完整序列:  [指令部分...] [### Response:] [回答部分...] [<eos>] [pad] [pad]
loss mask: [-100 -100 ...] [-100]       [计算loss...]  [计算]  [-100] [-100]
```

为什么 instruction mask 很重要? 数学上:

```
没有 mask: L = L_instruction + L_response    (指令部分的 loss 和回答部分的 loss 被平等对待)
有 mask:   L = L_response                     (梯度信号 100% 聚焦在回答质量上)
```

指令部分的 loss 让模型花力气学习"背诵指令", 这不是我们想要的能力。更糟糕的是, **当指令很长而回答很短时, 指令部分的 loss 会主导梯度, 淹没回答部分的信号**。

这在 Hugging Face TRL 的 `SFTTrainer` 中已是默认行为 (`dataset_text_field` + `response_template`), 也是工业界 SFT 的标准做法。

#### 第一个 EOS token 的保留: 小细节, 大影响

书中有一个精妙的设计: padding 中的第一个 `<|endoftext|>` 不被替换为 -100, 因为模型需要学会在回答结束后主动输出 EOS。如果 mask 掉了所有 EOS, 推理时模型不知道什么时候停下来, 会无限续写直到达到 max_length。

这体现了一个更普遍的原则: **训练和推理的一致性 (train-test consistency) 是微调中最容易被忽略但最致命的问题。** 训练时 mask 了什么, 推理时就"看不见"什么。

### 3.4 训练动态: 为什么 SFT 只需要 1-3 个 epoch

书中 SFT 只训练 2 个 epoch。这不是图快, 而是**训练更多反而有害**:

**灾难性遗忘 (Catastrophic Forgetting):** SFT 数据集很小 (数百到数万条), 多次遍历会让模型过度适配这些样本的模式, 同时"忘记"预训练时从数万亿 token 中学到的通用知识。具体表现为: 模型在 SFT 数据集上 loss 很低, 但在开放问题上回答质量下降、多样性降低、事实性出错。

**模式坍缩 (Mode Collapse):** 过度训练会让模型对所有问题给出模板化的回答, 丧失预训练时学到的表达多样性。

**业界经验:**
- Llama 2 的 SFT 阶段训练了 2 个 epoch
- InstructGPT 的论文中提到 SFT 只用了约 13K 条数据
- 许多团队报告 1 个 epoch 就是最优选择, 超过 3 个 epoch 几乎必然退化

**学习率的递减规律**也反映了同样的逻辑:

| 阶段 | 典型学习率 | 训练量 | 直觉 |
|---|---|---|---|
| 预训练 | 1e-4 ~ 6e-4 | 数 T tokens, 1 epoch | 从零开始, 大步走 |
| SFT | 1e-5 ~ 5e-5 | 数万条, 1-3 epoch | 轻推方向, 别破坏基础 |
| RLHF/DPO | 1e-6 ~ 5e-6 | 数万偏好对, 1 epoch | 微调旋钮, 只改品味 |
| 推理强化 (GRPO) | ~1e-6 | 数千到数万条, 1 epoch | 最轻的触碰, 涌现能力 |

每个阶段的学习率大约比前一阶段小一个数量级, 因为**越往后, 模型已经越"好", 改动的风险越大**。

### 3.5 评估: 一个至今未解决的难题

书中使用 LLM-as-Judge (Llama 3 8B 打分), 微调后的 GPT-2 Medium 得分 49.45/100。

**为什么评估如此困难?** 因为开放式文本生成是一个**病态问题 (ill-posed problem)**: 对同一个问题, "Paris" 和 "The capital of France is Paris, a city known for..." 都是正确答案, 但它们的 token 序列完全不同, 无法用 BLEU/ROUGE 等自动指标衡量。

**当前业界评估生态:**

| 方法 | 代表 | 核心思路 | 局限 |
|---|---|---|---|
| **标准化 benchmark** | MMLU, HellaSwag, ARC | 多选题, 有标准答案 | 只测知识和推理, 不测对话能力 |
| **LLM-as-Judge** | MT-Bench, AlpacaEval, Arena-Hard | 用强模型 (GPT-4) 给弱模型打分 | 评委偏见: 偏好长回答、格式化回答; 自我偏好 (GPT-4 给 GPT-4 打高分) |
| **人类评估** | Chatbot Arena (LMSYS) | 真实用户盲评两个模型的回答 | 成本高, 规模受限, 但被认为最可靠 |
| **任务完成率** | SWE-Bench (代码), GAIA (通用 Agent) | 让模型完成具体任务, 看能否成功 | 只适用于有明确成功标准的任务 |

**Chatbot Arena 的革命性意义**: LMSYS 的 Chatbot Arena 是目前业界最受认可的 LLM 评估方式。它让真实用户与两个匿名模型同时对话, 选择哪个更好, 然后用 Elo rating 排名。这绕开了"什么是好回答"的定义问题, 直接让人类偏好说话。截至 2025 年, 它已经收集了超过百万条人类投票, 是 LLM 排名的事实标准。

## 四、后训练的前沿: 书中没有覆盖的关键进展

### 4.1 从 RLHF 到 DPO: 偏好对齐的范式简化

SFT 之后, 模型能"听话"了, 但回答质量参差不齐。偏好对齐 (Alignment) 进一步教模型区分"好回答"和"坏回答"。

**RLHF (2022, InstructGPT/ChatGPT)** 的三阶段:

```
1. 收集人类偏好数据: 对同一个 prompt, 人类标注哪个回答更好
2. 训练 Reward Model: 拟合人类偏好, 给回答打分
3. 用 PPO (强化学习) 优化: 让 LLM 生成高分回答
```

RLHF 的问题: PPO 训练极不稳定, 超参敏感, reward hacking (模型学会骗 reward model 而非真正提高质量), 工程复杂度高 (需要同时运行 4 个模型: policy, value, reward, reference)。

**DPO (2023, Rafailov et al.)** 的关键洞察:

DPO 证明了一个数学恒等式: **最优的 RLHF 策略可以直接从偏好数据中推导出来, 不需要显式训练 reward model**。DPO 的损失函数:

```
L_DPO = -log sigmoid(beta * (log P_theta(y_w|x)/P_ref(y_w|x) - log P_theta(y_l|x)/P_ref(y_l|x)))

其中 y_w = 人类偏好的回答, y_l = 人类不偏好的回答, P_ref = SFT 后的参考模型
```

直觉: 增大好回答的概率, 减小坏回答的概率, 同时用参考模型约束不要偏离太远。

DPO 把 RLHF 从"训练 3 个模型的 RL 问题"简化为"训练 1 个模型的监督学习问题", 工程复杂度大幅降低。Llama 2, Zephyr, Mistral 等模型都采用了 DPO 或其变体。

**DPO 之后的演进:**

| 方法 | 年份 | 相对 DPO 的改进 |
|---|---|---|
| **IPO** (Identity PO) | 2023 | 修复 DPO 在偏好数据确定性高时的过拟合问题 |
| **KTO** | 2024 | 不需要成对比较, 只需要 thumbs up/down, 数据收集成本更低 |
| **ORPO** | 2024 | 把 SFT 和偏好对齐合并为一步, 不需要单独的 SFT 阶段 |
| **SimPO** | 2024 | 不需要参考模型, 用回答长度归一化替代 P_ref, 更简单 |

### 4.2 推理强化: DeepSeek-R1 与 GRPO 的突破

2024-2025 年最重要的后训练进展不在对话质量, 而在**推理能力**。

**OpenAI o1 (2024.09)** 首次展示: 通过让模型输出"思维链" (chain-of-thought), 可以在数学、编程等推理任务上获得巨大提升。但 o1 的训练方法未公开。

**DeepSeek-R1 (2025.01)** 开源了一种训练推理模型的方法, 核心是 **GRPO (Group Relative Policy Optimization)**:

```
传统 RLHF/DPO: 人类标注偏好 -> 模型学习"人觉得好的回答"
GRPO:          给模型一个数学题 -> 模型生成 N 个回答 -> 用规则判对错 -> 强化正确的, 惩罚错误的
```

GRPO 的关键创新:
1. **不需要 reward model**: 对数学题, 答案对错可以自动验证 (规则 reward)
2. **Group Relative**: 对同一个问题采样一组回答, 组内相对比较, 不需要绝对的 reward 分数
3. **涌现 "aha moment"**: 训练过程中模型自发学会了重新审视自己的推理、发现错误并纠正 -- 这是一种预训练和 SFT 都不具备的能力

这标志着后训练进入了一个新阶段: **不再只是"校准行为"或"优化品味", 而是通过 RL 真正涌现新能力**。

**完整的推理模型训练 pipeline (DeepSeek-R1):**

```
预训练 Base Model
  -> SFT (常规指令数据 + 少量 CoT 数据作为 cold start)
    -> GRPO (大量数学/代码/逻辑题, 规则 reward)
      -> 再次 SFT (用 GRPO 模型生成的高质量推理数据蒸馏回标准格式)
```

### 4.3 数据合成: SFT 数据从哪里来

书中用了 1100 条人工标注的数据。但在工业界, 高质量 SFT 数据的获取是一个核心竞争力:

**数据来源的演进:**

| 阶段 | 方法 | 代表 | 问题 |
|---|---|---|---|
| 人工标注 | 雇人写指令-回答对 | InstructGPT (40 名标注员) | 成本高, 规模受限, 标注员水平参差不齐 |
| 模型蒸馏 | 用强模型 (GPT-4) 生成数据, 微调弱模型 | Alpaca, Vicuna, WizardLM | 质量上限受限于 teacher 模型, 可能违反 ToS |
| Self-Instruct | 让模型自己生成指令和回答, 人工过滤 | Self-Instruct (2023) | 多样性和难度受限 |
| Evol-Instruct | 用 LLM 迭代地让指令变得更复杂 | WizardLM (2023) | 可能产生不自然的"人为复杂"指令 |
| **合成数据 pipeline** | 多阶段: 生成 -> 过滤 -> 验证 -> 增强 | UltraChat, Magpie (2024) | 当前主流, 兼顾质量和规模 |

**LIMA 论文的核心发现 (2023)**: 仅用 1000 条精选数据微调 LLaMA-65B, 效果接近用 52K 条数据微调的 Alpaca。这说明 **SFT 数据的质量远比数量重要**。但后续研究也发现, 对于小模型 (7B 以下), 数据量依然很关键 -- LIMA 的结论更适用于大模型, 因为大模型的预训练能力空间更完整, 需要的"行为校准"更少。

### 4.4 LoRA 及其演进: 参数高效微调的工程革命

书中的微调是全参数的 (或解冻最后几层)。业界的主流做法是 LoRA:

**LoRA 的核心思想 (2021):**

微调时, 权重的变化量 delta_W 是低秩的。所以不直接更新 W, 而是学习一个低秩分解:

```
W' = W + delta_W = W + A * B

其中 W: [d, d], A: [d, r], B: [r, d], r << d (通常 r=8~64)
可训练参数从 d^2 降到 2*d*r, 减少 100x+
```

**LoRA 之后的演进:**

| 方法 | 年份 | 改进 |
|---|---|---|
| **QLoRA** | 2023 | 将基础模型量化到 4-bit, LoRA 适配器保持 fp16, 24GB 显存即可微调 70B 模型 |
| **DoRA** | 2024 | 将权重分解为 magnitude + direction, 分别做 LoRA, 效果接近全量微调 |
| **LoRA+** | 2024 | 对 A 和 B 矩阵使用不同学习率 (B 的学习率更大), 收敛更快 |
| **rsLoRA** | 2024 | 调整 LoRA 的 scaling factor 为 1/sqrt(r) 而非 1/r, rank 增大时更稳定 |
| **GaLore** | 2024 | 在梯度空间做低秩投影而非权重空间, 更节省内存 |

**实践中的 LoRA 配置:**
```python
# 典型配置
lora_config = LoraConfig(
    r=16,                # rank, 越大越接近全量微调, 常用 8-64
    lora_alpha=32,       # scaling factor, 通常 = 2*r
    target_modules=[     # 对哪些模块注入 LoRA
        "q_proj", "k_proj", "v_proj", "o_proj",  # attention
        "gate_proj", "up_proj", "down_proj"       # FFN (新趋势: 也加 LoRA)
    ],
    lora_dropout=0.05,
)
```

早期 LoRA 只加在 attention 的 Q/V 矩阵上, 但 2024 年的经验表明加在所有线性层 (包括 FFN) 上效果更好, 这已成为新的默认做法。

### 4.5 Scaling 视角: 后训练的计算分配正在重新平衡

传统观点: 后训练是预训练的"附属品", 计算量占比不到 1%。但这个格局正在变化:

**Llama 3.1 405B (2024) 的后训练用了 6 轮迭代:**
```
SFT -> Rejection Sampling -> DPO -> SFT -> Rejection Sampling -> DPO
```
每轮都需要大量推理 (采样) + 训练, 后训练的总计算量已经不可忽略。

**DeepSeek-R1 的 GRPO 训练**: 需要对每个问题采样数十个回答, 然后做 RL 更新, 计算成本远超传统 SFT。

**趋势**: 后训练 (特别是 RL 阶段) 在总计算预算中的占比正在**从 <1% 向 10%+ 甚至更高**发展。这重新定义了"什么是训练一个 LLM" -- 不再是"预训练完成就结束", 而是"预训练只完成了一半, 后训练决定最终表现"。

## 五、两种微调的统一视角与深层对比

### 5.1 损失函数: 同一个 cross-entropy, 不同的信息量

```
分类微调:  CrossEntropy(model_output[-1], class_label)
           -> 一个样本只产生 1 个 loss 值 (最后一个 token 位置)
           -> 信息密度低: 每条数据只有 1 bit 左右的监督信号

指令微调:  sum_t CrossEntropy(model_output[t], input[t+1])
           -> 一个样本产生 T 个 loss 值 (每个 token 位置)
           -> 信息密度高: 每条数据有 T * log(vocab_size) 的理论监督信号
```

这解释了为什么书中指令微调用了更少的数据 (935 条) 却训练了更大的模型 (355M vs 124M) -- **每条 SFT 数据携带的梯度信号远多于分类标签**。

### 5.2 从"表示学习"到"策略学习"

深层来看, 分类微调和指令微调做的事情有本质区别:

- **分类微调**: 本质是**表示学习 (representation learning)** -- 学习一个好的 hidden state 空间, 使得不同类别可以被线性分开。重点在 transformer 内部的表示。
- **指令微调**: 本质是**策略学习 (policy learning)** -- 学习一个条件生成策略, 给定指令时输出合适的文本。重点在输出分布的形状。

这也解释了为什么分类微调需要改架构 (加分类头) 而指令微调不需要: 分类需要一个新的"读出器"来利用 hidden state, 而指令微调只需要调整已有的"生成器"的行为。

## 六、关键 Takeaways

1. **后训练的本质是"在能力空间中选择行为策略"**: 预训练建立了能力, 后训练决定了行为。能力是预训练的上限, 行为是后训练的输出。

2. **SFT 的杠杆效应惊人**: 935 条数据 + 2 个 epoch 就能让一个"只会续写"的模型变成"能遵循指令"的模型。这是因为 SFT 改变的是行为分布的模式, 而非能力的边界。但要注意这对大模型更成立, 小模型可能需要更多数据。

3. **数据工程 > 算法创新**: 在 SFT 中, 数据质量 (prompt 多样性、回答准确性、格式一致性) 的影响远大于训练算法的选择。padding/masking 策略、模板设计、loss mask 这些"脏活"直接决定训练效果。

4. **后训练的计算版图正在剧变**: 从 "SFT + 简单 RLHF" 到 "多轮迭代 SFT-DPO + RL 推理强化 (GRPO)", 后训练的复杂度和计算量都在急剧增长。DeepSeek-R1 证明了 RL 可以涌现预训练和 SFT 都无法获得的推理能力, 这将重塑后训练的投资分配。

5. **分类微调的历史定位**: 用 GPT 做分类是迁移学习时代的经典范式, 但在生产中, BERT 系列仍然是分类任务的首选 (更快, 更适合判别任务)。GPT 做分类的场景正在被 prompt-based 方法和 embedding + 轻量分类器替代。分类微调更多是理解"微调如何工作"的教学工具。

6. **评估是下一个待突破的瓶颈**: 当所有人都在优化模型, 但没有人能精确衡量"好了多少"时, 进步就变得盲目。Chatbot Arena 的人类评估是当前最可靠的标杆, 但它的带宽有限。如何低成本、高可靠地评估开放式生成质量, 是制约后训练发展的核心瓶颈之一。
